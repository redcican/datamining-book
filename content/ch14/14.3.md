# 14.3 数据需求分析与获取

§14.2 完成了项目目标的三层转化，产出了项目章程（定义 14.7）并通过了质量关口 G1。下一步自然的问题是：**支撑这些目标的数据从哪里来？够不够？质量如何？**

这正是 CRISP-DM 阶段 2（数据理解）和阶段 3（数据准备）的前半段——在动手清洗和特征工程之前，首先需要明确数据需求、评估数据源、验证数据质量。§1.3 中定义的 KDD 步骤 $$S$$（选择）和 $$P$$（预处理）的质量，根本上取决于这一阶段的工作深度。§2 全面介绍了数据预处理的技术手段；本节聚焦于**项目管理视角**——如何系统化地规划、获取和验证数据，确保后续的预处理和建模工作建立在可靠的数据基础上。

{% hint style="info" %}
📋 **本节学习目标**

完成本节学习后，读者应能够：

1. 编制结构化的数据需求清单，明确每个特征的来源、类型和优先级；
2. 运用多维度框架评估和选择数据源（质量、成本、时效、合规）；
3. 使用 §1.3 引入的五维度框架（完整性、唯一性、一致性、准确性、时效性）系统化地评估数据质量；
4. 估算项目所需的最小样本量，避免"数据不足"和"数据过多"两个极端；
5. 制定标签获取策略，理解不同标注方法的成本与质量权衡；
6. 编写规范的数据字典和数据质量报告，为质量关口 G2 做好准备。
{% endhint %}

**前置知识**：§14.2 问题定义与目标设定（项目章程），§1.3 数据质量五维度，§2.1 数据清洗技术。

## 14.3.1 数据需求清单

项目章程中的"数据概况"只是粗略的数据认知。在正式获取数据之前，需要将其细化为**结构化的数据需求清单**——这是连接"挖掘目标"与"实际数据"的桥梁。

{% block type="definition" %}
**定义 14.8**（数据需求清单）　数据需求清单是对项目所需数据的系统化描述，每条数据需求包含以下要素：

| 字段 | 说明 | 示例 |
|------|------|------|
| 特征名称 | 数据项的业务名称 | 交易金额 |
| 特征类型 | 数值/分类/时间/文本/二值 | 数值（连续） |
| 来源系统 | 数据存储位置 | 交易流水表 `t_transaction` |
| 获取方式 | SQL/API/文件/爬取 | SQL 查询 |
| 更新频率 | 实时/日/周/月/一次性 | 实时 |
| 历史深度 | 需要多长的历史数据 | 24 个月 |
| 优先级 | 必需/重要/可选 | 必需 |
| 预期作用 | 该特征在模型中的预期角色 | 反映消费能力，区分正常与异常交易 |
{% endblock %}

### 从挖掘目标反推数据需求

数据需求清单不是凭空拟定的，而是从 §14.2 确定的挖掘目标**反向推导**而来。以信用卡欺诈检测为例：

| 挖掘目标的组成部分 | 推导出的数据需求 |
|---|---|
| 目标变量："是否欺诈" | 需要历史欺诈标签（持卡人确认 + 银行调查结果） |
| 预测窗口："实时逐笔" | 需要交易级别的特征，而非日/月聚合 |
| 特征空间："交易行为模式" | 金额、时间、商户类别、地理位置、频率统计 |
| 上下文信息："用户画像" | 开卡时长、历史交易规模、信用额度 |
| 对比基线："历史正常模式" | 用户近 7/30/90 天的统计特征（均值、标准差、最大值） |

{% hint style="tip" %}
💡 **"先列清单，再看库存"**

一个常见的错误是先看手里有什么数据，然后围绕现有数据设计模型——这是"路灯效应"（只在灯光下找钥匙）。正确的做法是：先从业务目标出发列出**理想的数据需求清单**，然后逐项检查哪些数据可获取、哪些缺失需要补充、哪些需要替代方案。这样才能识别出真正的数据缺口，而非被现有数据的局限性束缚。
{% endhint %}

## 14.3.2 数据源评估与选择

大型组织中通常存在多个潜在的数据源——业务数据库、日志系统、第三方数据供应商、公开数据集等。选择正确的数据源是项目成功的前提。

### 四维度评估框架

{% block type="definition" %}
**定义 14.9**（数据源评估矩阵）　对每个候选数据源，从以下四个维度进行评估：

| 维度 | 评估内容 | 权重建议 |
|------|---------|---------|
| **质量** | 完整性、准确性、一致性（§1.3 五维度） | 40% |
| **可及性** | 获取难度、技术接口、权限审批流程 | 25% |
| **时效性** | 更新频率是否满足预测窗口需求 | 20% |
| **合规性** | 是否涉及个人信息、是否有使用授权 | 15%（一票否决） |

其中**合规性**虽然权重看似最低，但具有**一票否决权**——任何涉及未授权个人信息的数据源，无论质量多高，都不能使用。
{% endblock %}

### 内部数据 vs 外部数据

| 维度 | 内部数据 | 外部数据 |
|------|---------|---------|
| 典型来源 | 业务数据库、日志、CRM | 第三方数据供应商、公开数据集、API |
| 优势 | 与业务强相关，权限清晰 | 覆盖更广，补充内部盲区 |
| 劣势 | 可能存在数据孤岛 | 质量不可控，合规风险高 |
| 成本 | 获取成本低（已有基础设施） | 购买成本 + 对接开发成本 |
| 典型用法 | 核心特征（交易记录、用户行为） | 补充特征（天气、经济指标、舆情） |

**最佳实践**：先充分利用内部数据构建基线模型（MVM），评估基线性能后再决定是否引入外部数据。这样可以避免过早投入大量成本在外部数据上——很多时候，内部数据的深度挖掘（如特征工程）比引入新数据源更有效。

## 14.3.3 数据质量的系统化评估

§1.3 引入了数据质量的五个评估维度（完整性、唯一性、一致性、准确性、时效性），§2 详细介绍了各种清洗技术。本节从项目管理角度补充两个实践问题：**如何量化质量**和**何时判定"质量足够"**。

### 质量评估检查清单

将 §1.3 的五维度转化为可执行的检查清单：

| 维度 | 量化指标 | 计算方式 | 合格阈值（建议） |
|------|---------|---------|--------------|
| 完整性 $$\rho_c$$ | 非空比例 | $$1 - \frac{\text{缺失数}}{\text{总记录数}}$$ | ≥ 95%（关键特征 ≥ 99%） |
| 唯一性 $$\rho_u$$ | 去重后占比 | $$\frac{\text{唯一记录数}}{\text{总记录数}}$$ | ≥ 99%（主键字段 = 100%） |
| 一致性 $$\rho_k$$ | 格式/范围合规比 | $$\frac{\text{合规记录数}}{\text{总记录数}}$$ | ≥ 98% |
| 准确性 | 抽样验证正确率 | 人工校验 100–500 条样本 | ≥ 95% |
| 时效性 | 数据延迟 | 数据时间戳与当前时间的差值 | ≤ 预测窗口（如 ≤ 1 小时） |

### 质量评分卡

将五个维度的量化结果汇总为一个**综合质量评分**，用于快速判断数据集是否达到建模标准：

$$Q = \sum_{i=1}^{5} w_i \cdot q_i$$

其中 $$q_i \in [0, 1]$$ 是第 $$i$$ 个维度的归一化得分，$$w_i$$ 是对应权重（建议：完整性 0.30，准确性 0.25，一致性 0.20，时效性 0.15，唯一性 0.10）。$$Q \geq 0.85$$ 可作为"数据基本可用"的经验阈值。

{% hint style="warning" %}
⚠️ **质量阈值不是一刀切的**

上述阈值是通用的经验值，但不同项目的容忍度差异很大：

- **金融风控**：准确性要求极高（≥ 99%），因为错误的标签直接导致模型学到错误的模式；
- **推荐系统**：完整性要求可适当放宽（≥ 90%），因为用户行为数据天然稀疏；
- **实时预测**：时效性是硬约束，数据延迟超过预测窗口则毫无价值；
- **探索性分析**：各维度可适当放宽，但准确性仍不可妥协。

应根据项目的具体场景和 §14.2 中定义的业务目标来调整阈值。
{% endhint %}

## 14.3.4 样本量评估

"数据越多越好"是一个常见但不精确的认知。实际项目中，数据获取有成本（存储、清洗、标注），因此需要估算**最小可用样本量**——既确保模型有足够的统计功效，又避免不必要的资源浪费。

### 经验法则

对于传统机器学习模型（非深度学习），以下经验法则可以提供粗略估计：

| 任务类型 | 经验法则 | 说明 |
|---------|---------|------|
| 二分类 | 少数类 ≥ 10 × 特征数 | 如 50 个特征 → 少数类至少 500 条 |
| 多分类 | 每类 ≥ 20 × 特征数 | 如 50 个特征、5 类 → 每类至少 1,000 条 |
| 回归 | 总样本 ≥ 50 × 特征数 | 如 20 个特征 → 至少 1,000 条 |
| 聚类 | 总样本 ≥ 100 × 聚类数 | 如 5 个簇 → 至少 500 条 |

### 统计功效分析

更严谨的方法是基于统计检验的**功效分析**（power analysis），给定显著性水平 $$\alpha$$、期望的效应量 $$d$$ 和统计功效 $$1 - \beta$$，计算所需样本量。

对于两组均值差异的 t 检验，所需每组样本量为：

$$n \geq \frac{2(z_{\alpha/2} + z_\beta)^2}{d^2}$$

其中 $$d = \frac{|\mu_1 - \mu_2|}{\sigma}$$ 是 Cohen's d 效应量。典型设定为 $$\alpha = 0.05$$、$$\beta = 0.20$$（功效 80%），此时 $$z_{\alpha/2} + z_\beta \approx 2.80$$。

对于分类模型性能差异的检验（如"模型 A 的 AUC 是否显著优于模型 B"），可以使用 DeLong 检验的样本量公式——通常需要 500–2,000 个测试样本才能可靠地检测出 AUC 差异为 0.02–0.05 的改进。

### 不平衡场景的特殊考量

当正负类比例极端不平衡时（如 §13.2 的欺诈检测或 §13.6 的设备故障），样本量的瓶颈在**少数类**：

- 欺诈率 0.12% 意味着需要 ~83,000 笔交易才能获得 ~100 条正样本；
- 故障率 3.4% 意味着需要 ~3,000 条记录才能获得 ~100 条故障样本；
- 如果少数类样本不足，应在项目章程中明确标注为风险（§14.2 定义 14.7 第 7 项）。

## 14.3.5 标签获取策略

有监督学习的"监督信号"来自标签，而标签的获取往往是数据准备阶段**成本最高、质量最难保证**的环节。

### 五种标签来源

| 来源 | 成本 | 质量 | 延迟 | 适用场景 |
|------|------|------|------|---------|
| 系统日志 | 低 | 中高 | 低 | 点击/购买/流失等行为标签 |
| 业务规则 | 低 | 中 | 低 | 规则引擎已覆盖的已知模式 |
| 人工标注 | 高 | 高 | 高 | 文本分类、图像标注、复杂判断 |
| 众包标注 | 中 | 中低 | 中 | 大规模简单任务（如情感标注） |
| 弱监督 | 低 | 低中 | 低 | 标注函数生成噪声标签，后训练去噪 |

### 标签质量的影响

标签噪声对模型性能的影响远超大多数人的直觉。研究表明：

- **5% 的标签噪声**可使分类模型的准确率下降 2–5 个百分点；
- **10% 的标签噪声**可使模型学到系统性的错误模式；
- **标签噪声的影响不对称**：少数类的标签噪声比多数类更具破坏性——如果 100 条欺诈样本中有 10 条被错误标记为正常，等于丢失了 10% 的"监督信号"。

{% hint style="tip" %}
💡 **标签审计：成本最低的性能提升方式**

在模型性能瓶颈时，许多团队倾向于尝试更复杂的算法或更多的特征。但实践经验表明，**审计并修正标签质量**往往是投入产出比最高的改进方式：

1. 随机抽样 200–500 条训练样本，人工复核标签正确性；
2. 重点检查模型预测错误的样本——如果大量"错误预测"实际上是"标签错误"，说明标签质量是当前瓶颈；
3. 对不同标注来源的标签分别评估质量，优先修正低质量来源。
{% endhint %}

## 14.3.6 数据文档与质量关口 G2

数据获取阶段的最终产出是两份关键文档，也是质量关口 G2（定义 14.1）的审查对象：

### 数据字典

{% block type="definition" %}
**定义 14.10**（数据字典）　数据字典是对项目所用数据集中每个字段的结构化描述，至少包含以下信息：

| 元素 | 说明 | 示例 |
|------|------|------|
| 字段名 | 技术名称（数据库列名） | `txn_amount` |
| 业务含义 | 通俗描述 | 单笔交易金额（人民币，含小数） |
| 数据类型 | float/int/string/datetime/bool | float64 |
| 取值范围 | 合法值域 | [0.01, 999999.99] |
| 缺失率 | 空值占比 | 0.03% |
| 来源 | 数据表/API/计算字段 | `t_transaction.amount` |
| 更新频率 | 数据刷新周期 | 实时 |
| 备注 | 特殊说明（如业务规则、编码含义） | 退款交易金额为负值 |
{% endblock %}

### 数据质量报告

数据质量报告是对数据集整体质量状况的评估文档，包含：

1. **数据概况**：记录数、特征数、时间范围、文件大小；
2. **五维度评估**：每个维度的量化得分和综合评分 $$Q$$；
3. **问题清单**：已识别的质量问题（如特征 X 缺失率 12%、标签 Y 存在 3% 噪声）及处理方案；
4. **样本量评估**：少数类/多数类样本量是否满足建模需求；
5. **风险标注**：可能影响建模结果的数据局限性（如时间覆盖不足、分布偏移）。

### 质量关口 G2 的通过标准

质量关口 G2 的审查要点：

| 检查项 | 通过标准 | 未通过的后果 |
|--------|---------|------------|
| 数据字典完整 | 所有建模特征有完整描述 | 返回补充文档 |
| 综合质量评分 | $$Q \geq 0.85$$ | 返回进行数据清洗或寻找替代源 |
| 标签质量 | 人工抽检准确率 ≥ 95% | 返回进行标签审计和修正 |
| 样本量 | 少数类满足经验法则下界 | 评估是否补充数据或调整目标 |
| 合规审查 | 敏感字段已标注处理方案 | 返回进行脱敏或字段剔除 |

只有 G2 全部通过，项目才应正式进入建模阶段（§14.4）。

| 关键概念 | 说明 |
|---------|------|
| 数据需求清单 | 从挖掘目标反推的结构化数据需求描述 |

| 质量关口 G2 | 数据准备就绪的通过标准，确保建模基础可靠 |

## 习题

### A 级（概念理解）

**A1.** 解释"先列清单，再看库存"原则的含义。为什么从现有数据出发设计模型是一种"路灯效应"？

**A2.** 数据源四维评估中，合规性为什么具有"一票否决权"？举一个具体例子说明高质量但不合规的数据源为什么不能使用。

### B 级（分析推理）

**B1.** 在不平衡分类场景中（如欺诈率 0.1%），样本量的瓶颈为什么在少数类？如果总数据量为 100 万条但只有 1,000 条正样本，用 SMOTE 过采样到 1:1 能否替代获取更多真实正样本？分析两者的优劣。

**B2.** 标签噪声对少数类的影响为什么比多数类更严重？用信息论的观点（信号-噪声比）解释这一现象。

### C 级（方法应用）
