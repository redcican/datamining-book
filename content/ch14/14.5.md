# 14.5 模型部署与监控

实验报告通过质量关口 G4 后，模型进入 CRISP-DM 的阶段 6（部署）——也是 §14.1 中提到的"70% 项目折戟之处"。将一个在 Notebook 中表现优异的模型转化为稳定运行的生产服务，涉及的挑战从"算法"转向"工程"：推理延迟、系统可用性、资源成本、异常处理——这些在实验阶段几乎不用考虑的问题，在生产环境中成为决定项目成败的关键。

更重要的是，部署不是终点。§14.1 的 MLOps 框架（定义 14.3）强调了部署之后的**持续监控**——模型在真实环境中的性能会随时间衰退，只有及时检测并应对数据漂移和概念漂移，模型才能持续创造价值。

{% hint style="info" %}
📋 **本节学习目标**

完成本节学习后，读者应能够：

1. 根据业务场景选择合适的部署模式（批量推理 / 在线服务 / 流式处理）；
2. 使用上线前检查清单确保模型的生产就绪性；
3. 设计模型监控体系，覆盖数据质量、模型性能和系统健康三个维度；
4. 使用 PSI 和 KL 散度等指标检测数据漂移，设定合理的告警阈值；
5. 制定模型更新策略，平衡更新频率与运维成本。
{% endhint %}

**前置知识**：§14.1 MLOps 扩展生命周期（定义 14.3），§14.4 实验报告。

## 14.5.1 部署模式选择

不同的业务场景对模型推理的时效性要求差异巨大，对应三种主要的部署模式：

| 模式 | 推理时机 | 延迟要求 | 典型场景 | 技术实现 |
|------|---------|---------|---------|---------|
| **批量推理** | 定时（小时/天） | 无实时要求 | 营销名单筛选、信用评分月度更新 | Spark/Airflow 定时任务 |
| **在线服务** | 实时逐条 | < 50–200 ms | 欺诈检测、搜索排序、推荐 | REST API + 模型容器 |
| **流式处理** | 事件驱动 | < 1s | IoT 异常检测、实时风控 | Kafka + Flink + 嵌入模型 |

### 选择决策树

1. **业务能否接受小时级延迟？** → 是 → 批量推理（最简单，优先考虑）
2. **需要逐条实时响应？** → 是 → 在线服务
3. **需要处理连续数据流？** → 是 → 流式处理

{% hint style="tip" %}
💡 **从批量开始，按需升级**

许多团队过早地选择在线服务架构，导致工程复杂度远超必要。如果业务方说"希望尽快看到结果"，先确认这个"尽快"是"每天一次"还是"每秒一次"——前者用批量推理一个 Cron 任务就能解决，后者才需要在线服务的完整工程投入。
{% endhint %}

## 14.5.2 上线前检查清单

{% block type="definition" %}
**定义 14.13**（模型上线检查清单）　模型部署到生产环境前，须通过以下检查项：

| 类别 | 检查项 | 验证方法 |
|------|--------|---------|
| **功能正确性** | 生产环境预测结果与离线一致 | 用相同输入对比离线/在线预测结果 |
| **性能达标** | 推理延迟满足 SLA | 压力测试（P50/P95/P99 延迟） |
| **资源合理** | CPU/内存/GPU 使用量在预算内 | 资源监控 |
| **异常处理** | 缺失值/异常输入不导致服务崩溃 | 边界用例测试 |
| **回滚方案** | 出问题时能快速回退到上一版本 | 回滚演练 |
| **监控就绪** | 监控仪表盘和告警规则已配置 | 模拟触发告警 |
| **文档完整** | API 文档、依赖说明、运维手册 | 文档审查 |
| **合规审查** | 模型不使用禁止特征，可提供解释 | 合规团队签字 |
{% endblock %}

### 灰度发布与 A/B 测试

即使检查清单全部通过，也不建议直接全量上线。**灰度发布**策略将风险降到最低：

| 阶段 | 流量占比 | 持续时间 | 关注指标 |
|------|---------|---------|---------|
| 影子模式 | 0%（仅记录预测，不影响业务） | 1–2 周 | 预测分布与离线一致性 |
| 小流量测试 | 1–5% | 1 周 | 线上指标 vs 对照组 |
| 扩大灰度 | 10–50% | 1–2 周 | 统计显著性验证 |
| 全量上线 | 100% | — | 持续监控 |

## 14.5.3 模型监控体系

模型上线后，监控是确保其持续有效的唯一手段。监控体系覆盖三个层次：

### 三层监控架构

| 层次 | 监控对象 | 关键指标 | 告警条件 |
|------|---------|---------|---------|
| **数据层** | 输入数据的质量和分布 | 缺失率、PSI、特征分布偏移 | PSI > 0.25 或缺失率突增 |
| **模型层** | 预测质量和行为 | 预测分布偏移、业务指标（如转化率） | 指标连续 3 天下降超过阈值 |
| **系统层** | 服务的可用性和性能 | 延迟、错误率、吞吐量、资源使用 | P99 延迟 > SLA 或错误率 > 1% |

### 数据漂移检测

§14.1 已引入 PSI（定义 14.3）作为数据漂移的量化指标。在实际监控中，PSI 的使用有以下细节：

**分箱策略**：将特征值域分为 $$k$$ 个等频分箱（通常 $$k = 10$$），分别计算训练期和监控期在每个分箱中的比例 $$p_i$$ 和 $$q_i$$：

$$\text{PSI} = \sum_{i=1}^{k} (p_i - q_i) \ln \frac{p_i}{q_i}$$

**判断标准**：
- PSI < 0.10：分布无显著变化，无需行动
- 0.10 ≤ PSI < 0.25：轻微漂移，需要关注
- PSI ≥ 0.25：显著漂移，应触发模型评估或重训练

**多特征监控**：对每个输入特征分别计算 PSI，当**任一关键特征** PSI ≥ 0.25 或**多个特征**同时 PSI ≥ 0.10 时，触发告警。

{% block type="definition" %}
**定义 14.14**（模型监控指标体系）　生产模型的完整监控指标体系包含：

| 指标类别 | 具体指标 | 计算频率 | 告警阈值 |
|---------|---------|---------|---------|
| 数据完整性 | 输入缺失率 | 每批次 | > 5% |
| 数据漂移 | 特征 PSI | 每日/每周 | 单特征 > 0.25 |
| 预测漂移 | 预测值分布 PSI | 每日 | > 0.20 |
| 业务指标 | 如 CTR、转化率、拦截率 | 每日 | 连续 3 天下降 > 10% |
| 系统延迟 | P50/P95/P99 延迟 | 实时 | P99 > SLA |
| 系统错误 | HTTP 5xx 率 | 实时 | > 1% |
{% endblock %}

## 14.5.4 模型更新策略

当监控检测到性能衰退时，需要触发模型更新。更新策略的选择取决于漂移的类型和严重程度：

| 策略 | 触发条件 | 操作 | 适用场景 |
|------|---------|------|---------|
| 定期重训练 | 固定时间周期（如每月） | 用最新数据重训练同一模型 | 数据缓慢变化的场景 |
| 触发式重训练 | PSI > 0.25 或业务指标下降 | 自动触发训练管道 | 漂移不可预测的场景 |
| 在线学习 | 每条新数据到达时 | 增量更新模型参数 | 高频变化 + 实时要求 |
| 全面重建 | 重训练无法恢复性能 | 回到 §14.3 重新评估数据和特征 | 概念漂移（根本性变化） |

{% hint style="warning" %}
⚠️ **重训练 ≠ 重部署**

重训练产出的新模型必须经过与首次部署相同的验证流程（检查清单 + 灰度发布）。自动化的重训练管道如果没有配套的自动化验证，可能将一个有缺陷的新模型直接推入生产环境——这比不更新更危险。
{% endhint %}

## 本节小结

| 关键概念 | 说明 |
|---------|------|
| 部署模式 | 批量推理（最简单）/ 在线服务（实时）/ 流式处理（事件驱动） |
| 上线检查清单 | 功能、性能、异常处理、回滚、监控、文档、合规八项检查 |
| 灰度发布 | 影子模式 → 小流量 → 扩大 → 全量，渐进式降低风险 |
| 三层监控 | 数据层（PSI）+ 模型层（业务指标）+ 系统层（延迟/错误率） |
| PSI 阈值 | < 0.10 正常，0.10–0.25 关注，≥ 0.25 触发行动 |
| 更新策略 | 定期 / 触发式 / 在线学习 / 全面重建，按漂移类型选择 |

### A 级（概念理解）

**A1.** 解释批量推理、在线服务和流式处理三种部署模式的区别。为什么建议"从批量开始"？

**A2.** 什么是灰度发布？为什么即使上线检查清单全部通过，仍然不建议直接全量上线？

### B 级（分析推理）

**B1.** 某模型上线三个月后，PSI 从 0.05 逐渐上升至 0.30，但业务指标（转化率）尚未下降。应该立即重训练吗？分析 PSI 上升但业务指标未变的可能原因。

### C 级（方法应用）

**C1.** 为 §13.6 的预测性维护模型设计一个完整的监控方案：(a) 选择部署模式并说明理由；(b) 列出需要监控的关键指标及告警阈值；(c) 制定更新策略。

### D 级（编程实践）

**D1.** 实现一个 PSI 监控模块：输入训练期特征数据和当前期特征数据，输出每个特征的 PSI 值，并用红/黄/绿颜色标注漂移严重程度。用 §13.7 的 PM2.5 数据模拟：以 2010–2012 年为训练期，逐月计算 2013–2014 年的 PSI。

### E 级（案例分析）

**E1.** 按定义 14.13 的检查清单，评估 §13.2 的欺诈检测模型是否"部署就绪"。识别哪些检查项需要额外工作，并提出具体的补充方案。

### F 级（开放探讨）

**F1.** 讨论模型监控中的"告警疲劳"问题：如果告警阈值设得太灵敏，运维人员每天收到大量误报，最终可能忽略真正的告警。如何在"灵敏性"和"可操作性"之间找到平衡？（不少于 300 字。）
