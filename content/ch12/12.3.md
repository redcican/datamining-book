# 12.3 大数据挖掘平台（Hadoop, Spark）

§12.2 展示了 Python 生态系统在数据挖掘中的强大能力——scikit-learn 的统一接口、pandas 的灵活数据操作、Pipeline 的端到端建模流程。然而，这套工具链有一个根本性的局限：**它运行在单台机器上**。当数据量从 GB 级增长到 TB 甚至 PB 级时，单机的内存、磁盘 I/O 和 CPU 都将成为瓶颈。一个典型的例子：案例 12.2 中的电信客户数据仅有 7000 条记录，scikit-learn 可以在秒级完成训练；但如果这家运营商拥有 1 亿用户、每人数百条行为记录，数据总量达到数 TB，scikit-learn 甚至无法将数据加载到内存中。

**分布式计算框架**正是为解决这一规模挑战而设计的。它们将数据和计算分散到数十乃至数千台机器上，通过**数据分区**和**并行处理**将单机无法完成的任务变为可能。在数据挖掘领域，两个平台主导了分布式计算的格局：**Hadoop** 开创了大规模批处理的先河，**Apache Spark** 则以内存计算和对迭代算法的友好性成为当前的事实标准。

本节将从分布式计算的基本原理出发，分别剖析 Hadoop 和 Spark 的架构设计，深入介绍 Spark MLlib 的分布式机器学习能力，最后通过 PySpark 案例演示大规模数据挖掘的完整流程。

## 12.3.1 从单机到分布式

### 规模瓶颈

§12.1 的平台能力矩阵中，scikit-learn 在数据层的评分仅为"有限支持"（△）。这一局限源于 Python 单进程、单机的执行模型——pandas 的 `read_csv` 需要将整个文件加载到内存，scikit-learn 的 `fit` 在单个 CPU 上串行执行。下表展示了不同数据规模下单机工具的典型瓶颈：

| 数据规模 | 典型场景 | 单机瓶颈 | 解决方案 |
|---------|---------|---------|---------|
| < 1 GB | 学术数据集、原型开发 | 无 | scikit-learn / pandas |
| 1–10 GB | 中型企业数据 | 内存不足 | Polars / 采样 + scikit-learn |
| 10–100 GB | 大型企业日志 | 内存 + CPU | Dask / Spark |
| 100 GB–TB | 电商行为数据 | 完全不可行 | Spark / Hadoop |
| > TB | 互联网级数据 | 完全不可行 | Spark 集群 / 云服务 |

当数据量超过单机内存时，需要一种根本不同的计算范式——**将数据分散存储在多台机器上，将计算推送到数据所在的节点，最后汇总结果**。这就是分布式计算框架的核心思想。

{% block type="definition" %}
**定义 12.7**（分布式计算框架）　分布式计算框架是一种将大规模数据处理任务分配到多台计算节点上并行执行的软件系统，具备以下三大核心属性：

1. **数据分区**（Data Partitioning）：将数据集切分为多个分区（partition），分散存储在集群的不同节点上。每个节点仅持有数据的一个子集，从而突破单机内存限制。

2. **并行计算**（Parallel Computation）：每个节点独立处理其本地分区的数据，多个节点同时执行相同的操作。节点间通过网络交换中间结果（如 Shuffle 操作）以完成需要全局信息的计算。

3. **容错机制**（Fault Tolerance）：当某个节点发生故障时，框架能够自动检测失败并重新执行受影响的任务，无需重启整个计算过程。实现方式包括数据副本（HDFS）和计算血统（Spark RDD lineage）。

分布式框架的理论加速上限由 **Amdahl 定律**给出：

$$
S(N) = \frac{1}{(1 - P) + \frac{P}{N}} \tag{12.4}
$$

其中 $$N$$ 为并行节点数，$$P$$ 为可并行化的任务比例（$$0 \leq P \leq 1$$）。当 $$N \to \infty$$ 时，$$S \to \frac{1}{1 - P}$$——即使使用无限多的节点，加速倍数也受限于串行部分的比例。
{% endblock %}

式 (12.4) 揭示了一个重要的工程启示：分布式系统的性能瓶颈往往不在于节点数量，而在于**串行瓶颈**——数据的 Shuffle（跨节点传输）、全局聚合和同步等操作。Spark 相比 Hadoop MapReduce 的一大优势正是通过减少磁盘 I/O 和优化 Shuffle 来提高可并行化比例 $$P$$。

{% hint style="warning" %}
⚠️ **分布式并非万能**

分布式计算引入了显著的**额外开销**：数据序列化/反序列化、网络传输、任务调度和节点间同步。对于小数据集（< 10 GB），这些开销可能超过并行带来的收益，导致分布式方案反而比单机 scikit-learn 更慢。经验法则：**只有当数据无法装入单机内存时，才考虑分布式方案**。
{% endhint %}

## 12.3.2 Hadoop 生态系统

### HDFS 与 MapReduce

2006 年，Google 发表的 GFS 和 MapReduce 论文的开源实现——**Apache Hadoop**——开启了大数据处理的新纪元。Hadoop 的核心由两部分组成：**HDFS**（分布式文件系统）负责数据存储，**MapReduce**（计算框架）负责数据处理。

**HDFS**（Hadoop Distributed File System）将文件切分为固定大小的块（默认 128 MB），每个块冗余存储在 3 个节点上。这种设计提供了两个关键能力：(1) 突破单机磁盘容量限制，集群可存储 PB 级数据；(2) 任意节点故障不会导致数据丢失。

{% block type="definition" %}
**定义 12.8**（MapReduce 计算模型）　MapReduce 是一种面向批处理的分布式计算模型，将任务分解为两个阶段：

1. **Map 阶段**：每个数据分区上独立执行映射函数 $$f_{\text{map}}$$，将输入记录转换为中间键值对 $$(k, v)$$：
$$
f_{\text{map}}: \text{record} \to \{(k_1, v_1), (k_2, v_2), \ldots\}
$$

2. **Reduce 阶段**：所有具有相同键 $$k$$ 的中间值被汇集到同一个节点，由归约函数 $$f_{\text{reduce}}$$ 聚合为最终结果：
$$
f_{\text{reduce}}: (k, [v_1, v_2, \ldots]) \to \text{result} \tag{12.5}
$$

两阶段之间存在一个隐含的 **Shuffle** 步骤——将 Map 输出按键分组并跨网络传输到 Reduce 节点。Shuffle 通常是 MapReduce 作业的性能瓶颈，因为它涉及大量的磁盘 I/O 和网络通信。
{% endblock %}

MapReduce 的经典应用是**词频统计**：Map 阶段将每个单词映射为 `(word, 1)`，Reduce 阶段将同一单词的计数求和。然而，对于数据挖掘中的**迭代算法**（如 K-Means、梯度下降），MapReduce 的效率极低——每次迭代都需要将中间结果写入 HDFS 再重新读取，磁盘 I/O 成为致命瓶颈。

### Hadoop 生态全景

围绕 HDFS 和 MapReduce，一个庞大的生态系统逐步形成：

| 组件 | 功能 | 对应 §12.1 架构层 |
|------|------|------------------|
| **HDFS** | 分布式文件存储 | 数据层 |
| **YARN** | 集群资源管理与调度 | 数据层 |
| **Hive** | SQL 查询引擎（转换为 MapReduce） | 处理层 |
| **HBase** | 分布式列族数据库 | 数据层 |
| **Pig** | 数据流脚本语言 | 处理层 |
| **Mahout** | 分布式机器学习（已被 Spark 取代） | 建模层 |

<figure><img src="/figures/ch12/fig12_3_01_hadoop_spark_architecture.png" alt="Hadoop 与 Spark 架构对比" /><figcaption><strong>图 12.3.1</strong>　Hadoop 与 Spark 架构对比。左侧为 Hadoop 经典三层架构（HDFS → YARN → MapReduce），右侧为 Spark 架构（可插拔存储 → Driver/Executor → <em>Spark SQL / MLlib / Streaming / GraphX</em>）。中间标注三大关键差异：磁盘 I/O vs 内存计算、两阶段 vs DAG 执行引擎、批处理 vs 批+流。</figcaption></figure>

{% hint style="tip" %}
💡 **Hadoop 的历史地位**

尽管 Spark 在大多数场景下已取代 MapReduce 作为首选计算引擎，Hadoop 的贡献不应被低估：(1) HDFS 仍然是许多企业的底层存储标准，Spark 本身也常运行在 HDFS 之上；(2) YARN 作为通用资源管理器，同时支持 MapReduce、Spark、Flink 等多种计算框架；(3) Hadoop 生态中的 Hive 和 HBase 在数据仓库和实时查询领域仍有广泛应用。
{% endhint %}

## 12.3.3 Apache Spark：内存计算引擎

### 设计理念

2014 年，来自 UC Berkeley AMPLab 的 Apache Spark 正式成为 Apache 顶级项目，其核心创新在于**内存计算**——将中间结果缓存在内存中，避免了 MapReduce 每次迭代都需要读写 HDFS 的开销。对于需要数十乃至数百次迭代的机器学习算法，这一改进带来了 10–100 倍的性能提升。

{% block type="definition" %}
**定义 12.9**（弹性分布式数据集）　**RDD**（Resilient Distributed Dataset）是 Spark 的核心抽象，具有以下属性：

1. **不可变性**（Immutable）：RDD 一旦创建不可修改，任何变换操作生成新的 RDD。
2. **分区性**（Partitioned）：数据被划分为多个分区，分布在集群的不同节点上。
3. **血统追踪**（Lineage）：每个 RDD 记录其从父 RDD 派生的变换序列。当某个分区数据丢失时，Spark 可根据血统重新计算，实现无副本的容错。

RDD 上的操作分为两类：
- **转换**（Transformation）：惰性操作，定义新 RDD 但不立即执行（如 `map`、`filter`、`groupByKey`）；
- **动作**（Action）：触发实际计算并返回结果（如 `count`、`collect`、`saveAsTextFile`）。

Spark 将一系列转换编译为**有向无环图**（DAG），优化执行计划后以 Stage 为单位提交给集群执行。
{% endblock %}

### Spark SQL 与 DataFrame API

在实际数据挖掘中，直接操作 RDD 较为底层。Spark 2.0 引入的 **DataFrame API** 提供了类似 pandas 的高级接口，同时享受 Spark SQL 优化器（Catalyst）的自动优化：

```python
from pyspark.sql import SparkSession

spark = SparkSession.builder.appName("mining").getOrCreate()
df = spark.read.csv("hdfs:///data/logs.csv", header=True, inferSchema=True)
# 类 pandas 操作，但分布式执行
result = (df.filter(df["age"] > 30)
            .groupBy("city")
            .agg({"salary": "mean", "purchase": "sum"})
            .orderBy("avg(salary)", ascending=False))
result.show()
```

与 pandas 的关键差异在于：(1) DataFrame 的操作是**惰性求值**的——上述代码在 `show()` 被调用前不会执行任何计算；(2) Spark 的 Catalyst 优化器会自动重排操作顺序、合并过滤条件、选择最优的 Join 策略。

<figure><img src="/figures/ch12/fig12_3_02_spark_execution_model.png" alt="Spark 执行模型：RDD 血统、DAG 与内存优势" /><figcaption><strong>图 12.3.2</strong>　Spark 执行模型。(a) RDD 血统与 DAG：<em>textFile → flatMap → map → reduceByKey → collect</em> 转换链按 Shuffle 边界划分为两个 Stage，窄依赖管道化执行，宽依赖触发跨节点数据交换。(b) 迭代算法性能对比：MapReduce 每轮迭代需读写 HDFS（磁盘 I/O 瓶颈），Spark 将中间结果缓存在内存中，迭代算法可获得 10–100× 加速。</figcaption></figure>

{% hint style="tip" %}
💡 **pandas 用户的 PySpark 迁移指南**

| pandas 操作 | PySpark 等价 | 关键差异 |
|------------|-------------|---------|
| `df["col"]` | `df.select("col")` | PySpark 返回 DataFrame 而非 Series |
| `df[df.age > 30]` | `df.filter(df.age > 30)` | 惰性求值 |
| `df.groupby("x").mean()` | `df.groupBy("x").agg({"y": "mean"})` | 需显式指定聚合列 |
| `pd.merge(a, b, on="id")` | `a.join(b, "id")` | 自动分布式 Shuffle |
| `df.apply(func)` | `df.rdd.map(func)` 或 UDF | UDF 性能较低 |

核心原则：**用 DataFrame API 替代 RDD 操作**——Catalyst 优化器能对 DataFrame 做自动优化，但 RDD 操作是黑盒，无法优化。
{% endhint %}

## 12.3.4 Spark MLlib：分布式机器学习

### Pipeline API

Spark MLlib 提供了与 scikit-learn 高度类似的 **Pipeline API**——事实上，MLlib 的设计直接借鉴了 scikit-learn 的 Estimator/Transformer 模式（定义 12.5–12.6）。核心组件包括：

- **Transformer**：实现 `transform()` 方法，将一个 DataFrame 转换为另一个（如 `StringIndexer`、`VectorAssembler`、`StandardScaler`）；
- **Estimator**：实现 `fit()` 方法，从 DataFrame 中学习参数并返回一个 Transformer（如 `LogisticRegression`、`RandomForestClassifier`）；
- **Pipeline**：将多个 Transformer 和一个 Estimator 串联为端到端的工作流。

### 支持的算法

MLlib 涵盖了数据挖掘的核心算法，虽然种类不如 scikit-learn 丰富，但所有实现均为分布式版本：

| 类别 | 算法 | scikit-learn 对应 |
|------|------|------------------|
| 分类 | 逻辑回归、随机森林、GBT、SVM | ✓ 全部支持 |
| 回归 | 线性回归、随机森林回归、GBT 回归 | ✓ 全部支持 |
| 聚类 | K-Means、GMM、LDA（主题模型） | ✓ 全部支持 |
| 协同过滤 | ALS（交替最小二乘） | scikit-surprise |
| 特征工程 | TF-IDF、Word2Vec、PCA、MinMaxScaler | ✓ 全部支持 |
| 模型选择 | CrossValidator、TrainValidationSplit | GridSearchCV |

<figure><img src="/figures/ch12/fig12_3_03_mllib_pipeline_workflow.png" alt="Spark MLlib 分布式 Pipeline 工作流" /><figcaption><strong>图 12.3.3</strong>　Spark MLlib 分布式 Pipeline 工作流。从数据源（HDFS/S3）经 <em>StringIndexer → VectorAssembler → StandardScaler → RandomForestClassifier</em> 四级 Pipeline，到 <em>CrossValidator</em> 并行超参数搜索，最终持久化最优模型。底部标注与 scikit-learn Pipeline API 的对应关系（§12.2）。所有阶段均在集群上分布式执行。</figcaption></figure>

{% hint style="warning" %}
⚠️ **MLlib vs scikit-learn：不是替代，是互补**

MLlib 并非 scikit-learn 的替代品——两者适用于不同的数据规模：

1. **数据 < 10 GB**：scikit-learn 更快、算法更丰富、调试更方便；
2. **数据 10 GB – TB**：MLlib 是首选，或用 Spark 做预处理后采样到 scikit-learn；
3. **特殊算法需求**：scikit-learn 支持的算法种类远多于 MLlib（如 DBSCAN、t-SNE、HDBSCAN 等）。

许多生产系统采用**混合策略**：用 Spark 完成数据清洗和特征工程（TB 级），将结果采样或聚合到 GB 级后用 scikit-learn 做精细建模。
{% endhint %}

## 12.3.5 方法对比

下表从核心维度对比了主流分布式计算框架，帮助读者根据项目需求做出选择：

| 框架 | 计算模型 | 延迟 | 数据规模 | ML 支持 | 学习曲线 | 适用场景 |
|------|---------|------|---------|---------|---------|---------|
| **MapReduce** | 批处理 | 高 | PB 级 | Mahout（弱） | 高 | ETL、离线统计 |
| **Spark** | 批 + 流 | 中 | PB 级 | MLlib（强） | 中 | 通用数据挖掘 |
| **Dask** | 批处理 | 低–中 | TB 级 | Dask-ML | 低 | pandas/scikit-learn 扩展 |
| **Ray** | 批 + 流 | 低 | TB 级 | Ray Train | 中 | 深度学习分布式训练 |
| **Flink** | 流处理 | 极低 | PB 级 | FlinkML（弱） | 高 | 实时流计算 |

{% hint style="info" %}
📋 **快速选型建议**

- **已有 Hadoop 集群**：Spark on YARN——无需额外基础设施，直接读取 HDFS 数据；
- **Python 生态优先**：Dask——API 与 pandas/scikit-learn 几乎相同，迁移成本最低；
- **深度学习为主**：Ray——对 PyTorch 分布式训练的支持优于 Spark；
- **实时流处理**：Flink——事件驱动架构，端到端延迟毫秒级。
{% endhint %}

## 案例 12.3：PySpark 大规模用户行为分析

将本节介绍的 Spark 工具链串联起来，以下案例展示一个基于 PySpark MLlib 的分布式数据挖掘工作流。

{% block type="case" %}
**案例 12.3**（大规模用户行为分析）

**背景**：某电商平台积累了约 500 万条用户行为记录（浏览、加购、下单），包含用户属性（年龄段、会员等级、注册渠道）、行为统计（近 30 天浏览次数、加购次数、下单金额）和是否为活跃用户的标签。目标是构建分布式分类模型预测用户活跃度，为精准营销提供支撑。

**工具**：PySpark SQL（数据操作）、Spark MLlib（Pipeline + 建模 + 评估）。

```python
from pyspark.sql import SparkSession
from pyspark.ml import Pipeline
from pyspark.ml.feature import StringIndexer, VectorAssembler, StandardScaler
from pyspark.ml.classification import RandomForestClassifier, GBTClassifier
from pyspark.ml.evaluation import BinaryClassificationEvaluator
from pyspark.ml.tuning import CrossValidator, ParamGridBuilder
# --- 1. 创建 SparkSession 与数据加载 ---
spark = SparkSession.builder.appName("user_activity").getOrCreate()
df = spark.read.parquet("hdfs:///data/user_behavior.parquet")  # ~500万行
df.printSchema()  # age_group, vip_level, channel, views, carts, amount, label
# --- 2. 特征工程 ---
cat_indexers = [StringIndexer(inputCol=c, outputCol=c+"_idx")
                for c in ["age_group", "vip_level", "channel"]]
assembler = VectorAssembler(
    inputCols=["age_group_idx", "vip_level_idx", "channel_idx",
               "views", "carts", "amount"],
    outputCol="raw_features")
scaler = StandardScaler(inputCol="raw_features", outputCol="features")
# --- 3. 构建 Pipeline + 模型 ---
rf = RandomForestClassifier(labelCol="label", featuresCol="features",
                             numTrees=200, seed=42)
pipeline = Pipeline(stages=cat_indexers + [assembler, scaler, rf])
# --- 4. 交叉验证调参 ---
grid = (ParamGridBuilder()
        .addGrid(rf.numTrees, [100, 200])
        .addGrid(rf.maxDepth, [5, 10])
        .build())
evaluator = BinaryClassificationEvaluator(labelCol="label", metricName="areaUnderROC")
cv = CrossValidator(estimator=pipeline, estimatorParamMaps=grid,
                     evaluator=evaluator, numFolds=3, parallelism=4)
cv_model = cv.fit(df)
# --- 5. 评估与保存 ---
best_auc = evaluator.evaluate(cv_model.transform(df))
print(f"Best AUC: {best_auc:.4f}")
cv_model.bestModel.save("hdfs:///models/user_activity_rf")
# 典型结果: AUC ~0.87, 500万行训练约 8 分钟 (8 节点集群)
```

**结果分析**：在 8 节点 Spark 集群上，500 万行数据的 3 折交叉验证（含 4 种超参数组合）约 8 分钟完成，最优 AUC 约 0.87。相同数据量若用 scikit-learn 单机处理，仅数据加载就需要数分钟，完整的交叉验证可能需要数小时。值得注意的是 PySpark 的 Pipeline API 与 scikit-learn 高度相似——`StringIndexer` 对应 `LabelEncoder`，`VectorAssembler` 对应 `ColumnTransformer`，`CrossValidator` 对应 `GridSearchCV`——掌握 §12.2 内容的读者可以快速上手。`parallelism=4` 参数使 4 种超参数组合在集群上同时评估，进一步缩短调参时间。

## 本节小结

| 关键概念 | 说明 |
|---------|------|
| 分布式计算框架（定义 12.7） | 数据分区 + 并行计算 + 容错机制，Amdahl 定律约束加速上限 |
| MapReduce（定义 12.8） | Map → Shuffle → Reduce 两阶段模型，适合批处理但不适合迭代算法 |
| RDD（定义 12.9） | Spark 的核心抽象——不可变、分区、血统追踪，支持内存缓存 |
| Amdahl 定律（式 12.4） | $$S(N) = \frac{1}{(1-P) + P/N}$$，串行比例决定加速上限 |
| MapReduce 归约（式 12.5） | $$f_{\text{reduce}}: (k, [v_1, \ldots]) \to \text{result}$$ |
| Spark DataFrame | 类 pandas 高级接口，惰性求值 + Catalyst 自动优化 |
| Spark MLlib Pipeline | 借鉴 scikit-learn 的 Estimator/Transformer 模式，分布式执行 |

### B 级（分析推理）

**B1.** 使用 Amdahl 定律（式 12.4），计算以下场景的理论加速比：一个任务有 90% 可并行化（$$P = 0.9$$），分别使用 4、16、64 和 1024 个节点。讨论继续增加节点数的边际收益递减现象。

**B2.** Spark 的 DataFrame 操作是惰性求值的（直到 Action 触发才执行），而 pandas 是即时求值的。讨论惰性求值在以下两个场景中的优势和劣势：(a) 探索性数据分析（频繁查看中间结果）；(b) 大规模 ETL 管道（固定的处理流程）。

### C 级（比较分析）

**C1.** 对比 Spark MLlib 和 scikit-learn 的 Pipeline API：(a) 相似之处（Estimator/Transformer 模式、交叉验证）；(b) 差异之处（惰性求值、特征向量化方式、分布式执行）；(c) 各自的最佳适用场景。

**C2.** 对比 Spark、Dask 和 Ray 三个分布式框架在以下维度的表现：与 Python 生态集成度、学习曲线、对表格数据挖掘的支持、对深度学习的支持。

**D2.** 将案例 12.2（scikit-learn 客户流失预测）改写为 PySpark MLlib 版本。注意 API 的差异点：`ColumnTransformer` → `StringIndexer` + `VectorAssembler`，`cross_val_score` → `CrossValidator`。对比两版代码的结构相似性。

**E1.** 设计一个需要分布式处理的数据挖掘场景（数据量 > 50 GB）。描述：(a) 数据来源和规模；(b) 选择 Spark 还是 Dask，并论证理由；(c) 使用 Pipeline API 设计从数据加载到模型评估的完整流程。
