# 1.3 数据挖掘的流程与方法论

识别了挖掘任务的类型（1.2 节），并非意味着可以立即启动算法。现实的数据挖掘项目，绝非"加载数据、训练模型、导出结果"的线性三步流程——国际数据仓库研究所（TDWI）的调查显示，约 60%–85% 的数据挖掘项目因缺乏系统性方法论而未能在生产环境中产出预期价值（Pyle, 1999）。失败的根因往往不在于算法本身，而在于**业务问题定义不清、数据质量评估缺失、模型评估脱离业务目标，以及部署环节的忽视**。为此，学术界与工业界分别构建了若干结构化的流程框架，以规范化数据挖掘项目从问题定义到知识产出的完整生命周期。本节系统介绍其中最重要的两个框架——KDD 流程与 CRISP-DM，并建立数据类型与质量的评估体系，这是所有后续算法实践的"工程底座"。

{% hint style="info" %}
📋 **本节学习目标**

完成本节学习后，读者应能够：

1. 将 KDD 五步流程（选择→预处理→变换→挖掘→解释/评估）形式化为数据变换的复合链，并说明各步骤的输入、输出与目标；
2. 按照 CRISP-DM 六阶段框架（业务理解→数据理解→数据准备→建模→评估→部署），为给定业务场景划定每阶段的具体任务与交付物；
3. 对比 KDD、CRISP-DM、SEMMA 三大框架在阶段粒度、业务导向性与适用场景上的异同；
4. 区分结构化、半结构化、非结构化数据，并运用完整性、唯一性、一致性等五个维度评估数据质量；
5. 说明训练集、验证集、测试集划分的统计学依据，以及数据泄露（data leakage）的成因与预防措施。
{% endhint %}

**前置知识**：1.1 节（数据挖掘定义与 KDD 概念）；1.2 节（五类挖掘任务的形式化定义）。

## 1.3.1 结构化方法论的必要性

数据挖掘的工程实践与算法研究之间存在一道常被忽视的鸿沟。在学术环境中，研究者通常拿到预处理好的基准数据集（如 UCI Repository），直接评估算法性能。而在工业项目中，数据科学家面对的是分散于多个数据库的原始数据、不清晰的业务目标、利益相关方之间的分歧，以及将模型接入生产系统的工程挑战。

缺乏方法论的项目通常以以下方式失败：在业务目标尚未明确时便开始建模；在数据质量问题被忽视后发现模型学到了噪声规律；在评估阶段以学术指标（如准确率）代替业务指标（如 ROI）衡量成功；以及在模型交付后无人负责维护，数据分布漂移导致性能持续下降。

**结构化方法论的核心价值**在于：将隐性的工程经验显式化为可重复的操作规范，使不同背景的团队成员能够协调配合，并为项目管理提供里程碑与质量门（quality gate）。如同软件工程领域的 SCRUM 或 XP，数据挖掘领域的方法论框架既是技术指南，也是沟通语言。

## 1.3.2 KDD 流程：知识发现的操作性定义

KDD 框架由 Fayyad 等人（1996）在提出 KDD 定义的同一论文中给出，将知识发现形式化为一个**五步数据变换链**。这是将 1.1 节定义的抽象过程 $$\mathcal{D} \xrightarrow{\text{非平凡处理}} \mathcal{P}$$ 落地为具体操作步骤的首次尝试。

{% block type="definition" label="1.7" title="KDD 知识发现流程（Fayyad et al., 1996）" %}
设原始数据仓库为 $$\mathcal{D}$$，知识发现流程由如下五步数据变换复合而成：

$$
\mathcal{D} \;\xrightarrow{S}\; \mathcal{D}' \;\xrightarrow{P}\; \mathcal{D}'' \;\xrightarrow{T}\; \mathcal{D}''' \;\xrightarrow{M}\; f \;\xrightarrow{E}\; \mathcal{P}
\tag{1.10}
$$

各步骤定义：
- **$$S$$（选择，Selection）**：从 $$\mathcal{D}$$ 中选取与挖掘目标相关的目标数据集 $$\mathcal{D}' \subseteq \mathcal{D}$$，包括样本筛选与特征预选；
- **$$P$$（预处理，Preprocessing）**：应用清洗函数 $$P$$ 去除噪声、修复不一致、处理缺失值，得到 $$\mathcal{D}'' = P(\mathcal{D}')$$；
- **$$T$$（变换，Transformation）**：通过特征选择或降维映射 $$T: \mathbb{R}^d \to \mathbb{R}^{d'}$$（$$d' \leq d$$），将数据转化为适合挖掘的表示 $$\mathcal{D}''' = T(\mathcal{D}'')$$；
- **$$M$$（挖掘，Mining）**：应用学习算法 $$M$$ 从 $$\mathcal{D}'''$$ 提取候选模式 $$f = M(\mathcal{D}''')$$；
- **$$E$$（评估与解释，Evaluation/Interpretation）**：以兴趣度度量（interestingness measure）评估 $$f$$，结合领域知识筛选，输出有效知识集合 $$\mathcal{P}$$。
{% endblock %}

KDD 流程有两个关键特性，常被初学者忽视：

其一是**非线性性**。公式（1.10）呈现的线性外观具有误导性。实践中，步骤 $$E$$ 的评估结果可能揭示步骤 $$S$$ 的数据选择存在偏差，触发从 $$E$$ 回到 $$S$$ 的反馈循环；步骤 $$M$$ 的建模效果不佳，可能指向步骤 $$P$$ 或 $$T$$ 的缺陷，引发局部迭代。KDD 流程本质上是一个**在各步骤间反复循环直至满足质量标准**的迭代过程（如图 1.3.1 左侧的反馈箭头所示）。

其二是**步骤时间分布的严重不均**。Dasu 与 Johnson（2003）的研究发现，典型的数据挖掘项目中，步骤 $$S$$、$$P$$、$$T$$（数据准备）合计占用约 **80%** 的项目总工时，步骤 $$M$$（建模）仅占约 10%，步骤 $$E$$（评估与部署）占约 10%。这与初学者直觉中"建模是核心工作"的认知形成强烈反差，也揭示了数据工程能力的战略价值。

{% hint style="caution" %}
⚠️ **数据泄露（Data Leakage）：最常见的评估陷阱**

数据泄露是指在**训练阶段引入了模型在生产部署时无法获得的信息**，导致评估指标虚高而实际性能低下。在 KDD 框架中，泄露通常发生于步骤 $$P$$ 与 $$T$$：若归一化的均值和标准差在整个数据集（含测试集）上计算，测试集中的分布信息便"泄露"至训练流程，使得模型在测试集上的评估结果过于乐观。

**正确做法**：归一化参数（均值、方差）仅在训练集上 `fit()`，再应用于验证集和测试集 `transform()`。scikit-learn 的 `Pipeline` 类可自动保证这一约束，详见 1.2.7 节代码与第 2 章。
{% endhint %}

## 1.3.3 CRISP-DM：跨行业数据挖掘标准流程

CRISP-DM（Cross-Industry Standard Process for Data Mining）由 Wirth 与 Hipp（2000）提出，是目前工业界最广泛采用的数据挖掘项目管理框架。与 KDD 的技术视角不同，CRISP-DM 以**业务导向**为核心，将知识发现过程分解为六个阶段，并明确规定每个阶段的任务、输出物与阶段间的依赖关系（如图 1.3.1 所示）。

<figure>
<img src="/figures/ch01/fig1_3_01_kdd_crisp.png" alt="KDD 流程与 CRISP-DM 流程对比" style="max-width:100%;border-radius:6px;" />
<figcaption><strong>图 1.3.1</strong>　KDD 五步流程（左）与 CRISP-DM 六阶段框架（右）的阶段对应关系。中间连线标注了两框架在功能上的对应关系；左侧反馈箭头与右侧循环箭头均表达了迭代性这一核心特征。</figcaption>
</figure>

CRISP-DM 的六个阶段如下：

**阶段一：业务理解（Business Understanding）**　这是整个项目最关键的阶段，也是最容易被低估的阶段。该阶段的任务是将模糊的业务需求转化为精确的、可用数据验证的挖掘目标。具体产出包括：（1）业务目标陈述（如"将用户月流失率从 5% 降至 3%，对应增量收入约 500 万元"）；（2）数据挖掘成功标准的量化定义（如"在测试集上 AUC-ROC ≥ 0.85"）；（3）项目资源、约束与风险的评估。**业务目标的错误定义，是整个项目一切后续工作的"原罪"**——精确度最高的模型，若解决了错误的问题，其价值为零。

{% hint style="think" %}
🤔 **思考**：某银行希望"提高贷款部门的效率"。这是一个合格的数据挖掘目标吗？如何将其具体化为可建模的挖掘问题？（提示：效率的可量化定义是什么？提高效率的方式是减少审批时间、提高通过率、还是降低坏账率？这三个方向分别对应什么任务类型？）
{% endhint %}

**阶段二：数据理解（Data Understanding）**　在业务目标明确后，数据科学家需要**收集初始数据，进行探索性分析（EDA），评估数据质量**，并初步识别有趣的子集。这一阶段的产出是"数据理解报告"，描述数据的规模、特征分布、缺失模式、异常值以及数据源的可信度。本节 1.3.5 节将形式化定义数据质量的评估维度。

**阶段三：数据准备（Data Preparation）**　数据准备是 CRISP-DM 中最耗时的阶段，通常占据 60%–70% 的项目总工时。该阶段包括：（1）样本选择（哪些记录纳入分析）；（2）特征选择（哪些属性有价值）；（3）数据清洗（缺失值处理、异常值修复）；（4）特征构造（从已有特征衍生新特征）；（5）数据集成（跨来源数据的关联与合并）；（6）数据格式化（转换为建模算法所需格式）。数据准备阶段的质量直接决定建模阶段的上限——"垃圾进，垃圾出"（Garbage In, Garbage Out）是数据挖掘领域最古老也最有效的工程警示。

**阶段四：建模（Modeling）**　该阶段选择合适的建模技术，设计实验方案（包括数据划分策略与评估协议），训练并调整模型参数。值得注意的是，CRISP-DM 将"模型测试设计"（test design）明确列为建模阶段的任务，强调评估方案须在建模前确定而非事后选择——这直接杜绝了"事后挑选最佳评估指标"的结果操纵行为。

**阶段五：评估（Evaluation）**　阶段五区别于阶段四的关键在于：阶段四的评估是**技术性的**（在测试集上比较模型指标），阶段五的评估是**业务性的**（模型是否真正解决了阶段一定义的业务问题）。评估的产出是"继续/修订/放弃"的决策，以及对整个流程的回顾（process review）——常常此时才发现阶段一的问题定义需要修正，触发新一轮循环。

**阶段六：部署（Deployment）**　大量学术项目在模型训练完成后宣告结束，但在工业场景中，一个未部署的模型的商业价值为零。部署阶段包括：（1）制定部署计划（批量预测 vs. 实时在线推理）；（2）建立模型监控方案（检测数据漂移与性能下降）；（3）编制维护计划（定期再训练的触发条件）；（4）生成最终报告（模型卡、决策文档）。CRISP-DM 将循环箭头从阶段六指向阶段一，标志着"部署"不是终点，而是下一个改进周期的起点。

{% hint style="engineering" %}
🔧 **工程实践：CRISP-DM 的实际执行—— Sprint 化与里程碑管理**

在敏捷开发文化主导的互联网企业中，CRISP-DM 通常不作为严格的瀑布流程执行，而是以 **2–4 周的 Sprint** 为单位迭代。典型的 Sprint 规划如下：

- **Sprint 1–2**（业务理解 + 数据理解）：产出物包括业务目标文档、数据可用性评估报告、数据质量扫描结果；里程碑：项目立项评审通过。
- **Sprint 3–5**（数据准备）：产出物包括特征工程 Pipeline、数据集版本（含数据字典）；里程碑：训练数据集 Frozen。
- **Sprint 6–8**（建模 + 评估）：产出物包括模型实验记录（MLflow/Neptune.ai）、基线对比报告；里程碑：模型性能达到预定阈值，业务方评审通过。
- **Sprint 9–10**（部署 + 监控）：产出物包括 API 服务、监控面板（Grafana）、运维文档；里程碑：上线发布，灰度流量测试通过。

**关键工具链**（2025 年主流）：数据版本管理（DVC / Delta Lake）、实验追踪（MLflow）、模型服务（FastAPI / Triton Inference Server）、监控（Evidently AI / WhyLabs）。掌握这条工具链，是数据科学家从"学术玩家"转变为"工业实践者"的重要门槛。
{% endhint %}

## 1.3.4 三大方法论框架的比较：KDD、CRISP-DM 与 SEMMA

除 KDD 与 CRISP-DM 之外，SAS Institute 提出的 **SEMMA**（Sample, Explore, Modify, Model, Assess）是另一个在工业界有一定影响力的框架。SEMMA 较为简洁，更贴近工具操作层面，但缺乏对业务理解和部署阶段的显式定义。下表系统比较三个框架的异同：

| 维度 | KDD（1996） | CRISP-DM（2000） | SEMMA（SAS, 1998） |
|---|---|---|---|
| **阶段数** | 5 | 6 | 5 |
| **业务导向** | 弱（技术为主） | 强（业务目标驱动） | 中（工具操作为主）|
| **覆盖范围** | 数据 → 知识 | 问题定义 → 生产部署 | 数据采样 → 模型评估 |
| **迭代性** | 有（步骤间反馈） | 强（明确的循环箭头）| 有（评估后可回溯）|
| **部署阶段** | 隐含于"解释" | 明确独立阶段 | 无 |
| **维护/监控** | 不涵盖 | 部分涵盖 | 不涵盖 |
| **典型使用者** | 研究论文 | 工业项目 | SAS 工具用户 |

三个框架并非竞争关系，而是各有侧重。KDD 框架提供了数据变换的形式化语言（公式 1.10），适合理解挖掘过程的数学本质；CRISP-DM 提供了业务语境下的项目管理规范，是工业界项目沟通的通用语言；SEMMA 是 SAS 平台的操作指南，在 SAS 用户群体中普及。本书后续章节以 CRISP-DM 的框架组织代码实践，但技术推导遵循 KDD 的形式化符号体系。

{% hint style="supplement" %}
🔬 **补充：现代 MLOps 对方法论框架的扩展**

CRISP-DM 提出于 2000 年，彼时机器学习模型的生命周期管理远比今日简单。随着深度学习与大规模数据流处理的普及，MLOps（Machine Learning Operations）在 CRISP-DM 基础上延伸出若干新的关注点：

- **模型版本控制**：与软件 Git 类似，使用 DVC 对数据集版本和模型权重进行版本管理；
- **持续训练（Continuous Training）**：在数据分布漂移检测到后自动触发再训练流程；
- **在线 A/B 评估**：将模型的业务评估从离线测试集扩展到线上真实流量对比实验；
- **模型治理（Model Governance）**：满足 GDPR/《个人信息保护法》要求的可解释性与可审计性文档。

感兴趣的读者可参考 Google 的 ML 工程实践指南（Sculley et al., 2015; Breck et al., 2017）。
{% endhint %}

## 1.3.5 数据类型与数据质量评估

KDD 流程的步骤 $$S$$（选择）与步骤 $$P$$（预处理）的质量，根本上取决于**对数据性质的准确理解**。本节建立两个基础框架：数据类型分类体系与数据质量评估维度。

### 数据类型层次

现代组织中的数据，按其格式化程度可分为三大类（如图 1.3.2 所示）：

<figure>
<img src="/figures/ch01/fig1_3_02_data_types.png" alt="数据类型层次图" style="max-width:100%;border-radius:6px;" />
<figcaption><strong>图 1.3.2</strong>　数据类型层次图。从左至右，数据的格式化程度递减：结构化数据（关系表、时间序列）直接对应样本矩阵表示；半结构化数据（JSON/XML/图数据）需解析后建模；非结构化数据（文本、图像、音频）需要表示学习方能转化为向量。</figcaption>
</figure>

**结构化数据（Structured Data）**　以预定义的 Schema 组织，通常存储于关系型数据库（MySQL、PostgreSQL）或数据仓库（Hive、BigQuery）中。每条记录对应样本矩阵的一行，每个字段对应一个特征维度，可直接建立样本矩阵 $$\mathbf{X} \in \mathbb{R}^{n \times d}$$。时间序列是结构化数据的特殊形态，以时间戳为主键，特征值随时间演变，需要专门的时序挖掘方法（见第 7 章）。

**半结构化数据（Semi-structured Data）**　具有一定的组织结构（如 JSON 的键值层次、XML 的标签嵌套），但不强制要求固定 Schema——不同记录可含有不同字段集合。典型场景包括 Web API 响应（JSON）、系统日志（Log Files）以及图/网络数据（Graphs）。图数据可形式化为二元组 $$G = (V, E)$$，其中 $$V$$ 为节点集（如用户、商品），$$E$$ 为边集（如购买关系、转账行为），是社交网络分析与知识图谱挖掘的基础数据结构（见第 9 章）。

**非结构化数据（Unstructured Data）**　无预设格式的原始数据，包括自然语言文本、图像、音频与视频。非结构化数据占企业总数据量的约 65%，其处理的核心挑战是**表示学习**（representation learning）——将高维原始信号映射为低维稠密向量，使得语义上相似的内容在向量空间中距离相近。文本表示（TF-IDF、词嵌入、BERT）将在第 8 章系统介绍，图像表示（CNN 特征提取）将在第 10 章涉及。

{% hint style="mathbg" %}
📐 **数学基础：图数据的矩阵表示**

图 $$G = (V, E)$$（$$|V| = n$$）的核心矩阵表示为**邻接矩阵**（adjacency matrix）$$\mathbf{A} \in \{0,1\}^{n \times n}$$，其中 $$A_{ij} = 1$$ 当且仅当节点 $$i$$ 与节点 $$j$$ 之间存在边。对于无权图，$$\mathbf{A}$$ 为二值矩阵；对于带权图，$$A_{ij}$$ 存储边的权重。节点度（degree）向量 $$\mathbf{d} = \mathbf{A}\mathbf{1}$$，其中 $$\mathbf{1}$$ 为全一向量。**度矩阵** $$\mathbf{D} = \text{diag}(\mathbf{d})$$ 与邻接矩阵共同构成**图拉普拉斯矩阵** $$\mathbf{L} = \mathbf{D} - \mathbf{A}$$，是图信号处理与图神经网络（GNN）的数学核心（见第 9 章）。
{% endhint %}

### 数据质量评估框架

数据质量低劣是 KDD 步骤 $$P$$（预处理）存在的根本原因。为定量评估数据集的可靠性，学界通常采用以下五个维度（Batini et al., 2009；ISO 8000 标准）：

{% block type="definition" label="1.8" title="数据质量五维度" %}
设数据集为 $$\mathcal{D} = \{\mathbf{x}_i\}_{i=1}^n$$，$$\mathbf{x}_i \in (\mathcal{X}_1 \times \mathcal{X}_2 \times \cdots \times \mathcal{X}_d)$$，定义以下五个数据质量度量：

**（1）完整性（Completeness）**：数据集中非缺失值所占比例：

$$
\rho_c = 1 - \frac{\text{缺失值总数}}{n \times d} \in [0, 1]
\tag{1.11}
$$

**（2）唯一性（Uniqueness）**：数据集中非重复记录所占比例：

$$
\rho_u = 1 - \frac{\text{重复记录数}}{n} \in [0, 1]
\tag{1.12}
$$

**（3）一致性（Consistency）**：符合预定约束条件（如 $$0 \leq \text{年龄} \leq 150$$，$$\text{出生日期} < \text{入职日期}$$）的记录比例：

$$
\rho_k = \frac{\text{满足所有约束的记录数}}{n} \in [0, 1]
\tag{1.13}
$$

**（4）准确性（Accuracy）**：与外部参考数据源（ground truth）相符的记录比例（当参考数据源可获取时）。

**（5）时效性（Timeliness）**：数据记录与业务决策时间窗口的匹配程度，通常以最大可接受数据龄（data age）$$\tau_{\max}$$（天）量化。
{% endblock %}

在实际项目中，数据质量评估应在 CRISP-DM 阶段二（数据理解）完成，形成"数据质量报告"，并将低质量数据的处理策略纳入阶段三（数据准备）的计划中。

{% hint style="caution" %}
⚠️ **数据质量问题的类型与修复策略**

| 问题类型 | 描述 | 常见原因 | 修复策略 |
|---|---|---|---|
| **缺失值（Missing）** | $$x_{ij}$$ 为空/NaN | 传感器故障、用户未填写 | 删除/均值填充/KNN 填充/建模填充（见第 2 章）|
| **噪声（Noise）** | 测量误差导致的随机偏差 | 传感器精度限制 | Z-score/IQR 滤波、平滑 |
| **不一致（Inconsistency）** | 跨数据源同一实体数值矛盾 | 多系统异构、ETL 错误 | 实体对齐、业务规则修正 |
| **重复（Duplicate）** | 同一实体多条记录 | 多次录入、系统合并 | 精确/近似去重 |
| **异常值（Outlier）** | 统计上显著偏离正常范围 | 录入错误或真实极端值 | 需区分"错误"与"有意义的异常"——后者是异常检测的目标 |
{% endhint %}

### 数据集划分：训练集、验证集与测试集

数据质量评估完成后，在 CRISP-DM 阶段三（数据准备）的末尾，须将数据集划分为三个不相交的子集：

{% block type="definition" label="1.9" title="数据集三分法（Hold-out 划分）" %}
设完整标注数据集 $$\mathcal{D} = \{(\mathbf{x}_i, y_i)\}_{i=1}^n$$。**Hold-out 划分**将 $$\mathcal{D}$$ 随机分为三个不相交的子集：

$$
\mathcal{D} = \mathcal{D}_{train} \cup \mathcal{D}_{val} \cup \mathcal{D}_{test}, \quad
\mathcal{D}_{train} \cap \mathcal{D}_{val} = \mathcal{D}_{val} \cap \mathcal{D}_{test} = \mathcal{D}_{train} \cap \mathcal{D}_{test} = \emptyset
\tag{1.14}
$$

各子集用途：
- **训练集** $$\mathcal{D}_{train}$$（通常占 60%–70%）：用于拟合模型参数；
- **验证集** $$\mathcal{D}_{val}$$（通常占 10%–20%）：用于超参数调整与模型选择；
- **测试集** $$\mathcal{D}_{test}$$（通常占 15%–20%）：仅在最终评估时使用一次，报告模型的无偏泛化性能。

**核心约束**：测试集在整个开发过程中必须完全封存（not touched），直至所有模型选择决策完成后方可使用一次。违反此约束将导致测试集信息以间接方式泄露至模型，使最终性能评估不可信。
{% endblock %}

当样本量较小（$$n < 1000$$）时，Hold-out 划分的统计方差较大。此时通常采用 **$$k$$ 折交叉验证**（$$k$$-fold cross-validation）：将数据随机均分为 $$k$$ 个子集（通常 $$k=5$$ 或 $$k=10$$），依次以其中一个子集为验证集、其余 $$k-1$$ 个为训练集，循环 $$k$$ 次并报告平均性能与标准差。交叉验证的统计理论与实现细节将在第 3 章系统展开。

## 1.3.6 案例：电信客户流失预测——CRISP-DM 完整实践

{% block type="case" label="1.3" title="电信客户流失预测（IBM Telco Churn Dataset）" %}

**阶段一：业务理解**

某移动通信运营商面临月均客户流失率 5.2%（行业平均 2.5%）的严重问题。业务分析表明，挽留一个现有客户的成本约为获取新客户的 1/5。因此，若能提前 30 天识别出高流失风险客户并主动干预（如发放个性化优惠套餐），则预期可将流失率降至 3% 以下，年度挽留价值约 1800 万元。

**挖掘目标**：构建二分类模型，在客户发生实际流失前 30 天，识别流失概率 $$P(\text{流失} \mid \mathbf{x}) > 0.5$$ 的高风险客户群。

**业务成功标准**：AUC-ROC ≥ 0.85，且在 Precision ≥ 0.70 的约束下，Recall ≥ 0.60（即在控制误报率的前提下，至少捕获 60% 的即将流失客户）。

**阶段二：数据理解**

**IBM Telco Customer Churn Dataset**（IBM, 公开数据集）包含 $$n = 7{,}043$$ 位客户、$$d = 20$$ 个特征及二元标签（`Churn: Yes/No`）。流失率约 26.5%（正负样本比约 1:3）。

| 特征类别 | 典型特征 | 类型 |
|---|---|---|
| **服务使用** | 月费用（元）、合同类型（月/年/两年）| 数值型/有序类别型 |
| **产品订阅** | 是否订购网络安全、流媒体等增值服务 | 二值型（Yes/No）|
| **客户特征** | 在网时长（月）、是否有家属 | 数值型/二值型 |
| **账单** | 总费用、付款方式 | 数值型/名义类别型 |

EDA 发现：月租费用高（> 80 元/月）、合同类型为月付、在网时长短（< 12 个月）的客户流失率显著高于均值，符合业务直觉。

**阶段三：数据准备**

主要工作包括：（1）数值化：`TotalCharges` 列存在 11 条非数值记录（空字符串），识别为缺失值，以同组均值填充；（2）编码：名义类别特征（如付款方式）进行独热编码（One-Hot Encoding），有序特征（合同类型）进行有序编码；（3）标准化：连续数值特征（月费用、在网时长）进行 Z-score 标准化；（4）数据集划分：按 7:1.5:1.5 比例划分训练/验证/测试集，分层抽样（stratified split）保持各集合中流失率一致。

**阶段四：建模**

对比三种算法在验证集上的性能（超参数通过网格搜索确定）：

| 算法 | AUC-ROC | Precision | Recall | F1 |
|---|---|---|---|---|
| 逻辑回归（基线） | 0.837 | 0.651 | 0.578 | 0.612 |
| 随机森林 | 0.856 | 0.687 | 0.623 | 0.654 |
| XGBoost | **0.873** | **0.712** | **0.648** | **0.679** |

XGBoost 在所有指标上优于基线，且满足业务成功标准，作为最终候选模型。

**阶段五：评估**

在封存的测试集上运行 XGBoost，最终性能：AUC-ROC = 0.869，Precision = 0.704，Recall = 0.641，满足业务预设阈值。

阶段五评审发现：**高月费 + 月付合同**是最强预测信号（SHAP 值分析），但这类客户同时也是**高价值客户**（月消费 > 100 元）——流失挽留策略应优先为其提供长期合同折扣，而非通用优惠券，需与业务团队重新对齐干预策略（触发阶段一的局部迭代）。

**阶段六：部署**

模型以 FastAPI 包装为 REST API，每日凌晨批量对次日到期合同的客户评分，输出流失概率；CRM 系统自动对概率 > 0.5 的客户生成工单，分配给客服团队主动外呼。部署监控（Evidently AI）每周自动检测特征分布漂移，漂移超过阈值时触发再训练告警。

**思考延伸**　本案例中，26.5% 的流失率已属中度不平衡。若流失率仅为 2%（极端不平衡），模型训练与评估策略应如何调整？（提示：过采样/欠采样/代价敏感学习——见第 3 章。）
{% endblock %}

## 1.3.7 实验：KDD 五步流程的完整 Python 实现

本节以经典的鸢尾花分类问题为"模型项目"，逐步对应 KDD 五步流程，演示从原始数据到最终评估的完整代码路径。代码结构刻意与 KDD 的五个步骤一一对应，以便读者建立"代码段落 → 流程阶段"的直觉映射。

```python
# ============================================================
# 代码 1.3  KDD 五步流程：鸢尾花分类完整示例
# 数据集：UCI Iris（Fisher, 1936）
# 目标：演示 KDD 选择→预处理→变换→挖掘→评估 全流程
# 依赖库：numpy, pandas, scikit-learn
# ============================================================
import numpy as np
import pandas as pd
from sklearn import datasets
from sklearn.model_selection import train_test_split, cross_val_score
from sklearn.preprocessing import StandardScaler
from sklearn.decomposition import PCA
from sklearn.tree import DecisionTreeClassifier
from sklearn.metrics import (classification_report, confusion_matrix,
                              accuracy_score)
import warnings
warnings.filterwarnings("ignore")

print("=" * 60)
print("  KDD 流程演示：鸢尾花三分类（Iris Dataset）")
print("=" * 60)

# ─────────────────────────────────────────────────────────────
# 步骤 S：数据选择（Selection）
# 从原始"数据库"（sklearn 内置数据集）中
# 选取目标数据集，明确特征与标签。
# ─────────────────────────────────────────────────────────────
print("\n【步骤 S：数据选择】")

iris = datasets.load_iris()
X_raw = pd.DataFrame(iris.data, columns=iris.feature_names)
y = iris.target                                    # {0, 1, 2} 三类

print(f"原始数据规模：n={len(X_raw)} 个样本，d={X_raw.shape[1]} 个特征")
print(f"特征列表：{list(X_raw.columns)}")
print(f"标签分布：{dict(zip(*np.unique(y, return_counts=True)))}")
# 期望输出：{0: 50, 1: 50, 2: 50}  →  均衡三分类

# 特征预选：本例保留全部 4 个特征（实际项目中可基于领域知识删除低价值特征）
X_selected = X_raw.copy()
print(f"选取后特征数：{X_selected.shape[1]}")

# ─────────────────────────────────────────────────────────────
# 步骤 P：数据预处理（Preprocessing）
# 处理缺失值、检查重复记录、验证数值合法性。
# Iris 数据集较为清洁，此步骤以"空操作"示意流程完整性。
# ─────────────────────────────────────────────────────────────
print("\n【步骤 P：数据预处理】")

# P.1 缺失值检测
missing = X_selected.isnull().sum()
print(f"各特征缺失值数量：\n{missing.to_string()}")
# 期望输出：全部为 0（Iris 无缺失值）

# P.2 重复记录检测
n_dup = X_selected.duplicated().sum()
print(f"重复记录数：{n_dup}")
# Iris 中存在极少量重复（相同测量值），若需严格去重可执行 drop_duplicates()

# P.3 数值合法性验证（萼片/花瓣尺寸应大于 0）
assert (X_selected.values > 0).all(), "检测到非正数值，请检查数据源"
print("数值合法性验证通过（所有特征值 > 0）")

# ─────────────────────────────────────────────────────────────
# 步骤 T：数据变换（Transformation）
# T.1 数据集划分（Hold-out，定义 1.9）
# T.2 特征标准化（仅在训练集 fit，避免数据泄露）
# T.3 可选：PCA 降维（将 4 维压缩至 2 维可视化）
# ─────────────────────────────────────────────────────────────
print("\n【步骤 T：数据变换】")

# T.1 三分法划分（注意：先分割再拟合变换参数）
X_temp, X_test, y_temp, y_test = train_test_split(
    X_selected.values, y, test_size=0.15, random_state=42, stratify=y)
X_train, X_val, y_train, y_val = train_test_split(
    X_temp, y_temp, test_size=0.176, random_state=42, stratify=y_temp)
# test_size=0.176 使验证集约占原始数据的 15%（0.15 = 0.85 * 0.176）

print(f"训练集大小：{len(X_train)}  验证集：{len(X_val)}  测试集：{len(X_test)}")

# T.2 Z-score 标准化（fit 仅在训练集上）
scaler = StandardScaler()
X_train_s = scaler.fit_transform(X_train)   # fit + transform
X_val_s   = scaler.transform(X_val)          # 仅 transform（无 fit）
X_test_s  = scaler.transform(X_test)         # 仅 transform（无 fit）

print(f"训练集均值（标准化后）：{X_train_s.mean(axis=0).round(4)}")
# 期望输出：近似全零向量（标准化正确）
print(f"训练集标准差（标准化后）：{X_train_s.std(axis=0).round(4)}")
# 期望输出：近似全一向量

# T.3 PCA（可选，仅用于可视化说明；建模仍使用原始 4 维特征）
pca = PCA(n_components=2)
X_train_pca = pca.fit_transform(X_train_s)
explained = pca.explained_variance_ratio_
print(f"PCA 前 2 主成分方差解释率：{explained.round(4)}（累计 {explained.sum():.3f}）")
# 通常前 2 个主成分即可解释 >95% 的方差

# ─────────────────────────────────────────────────────────────
# 步骤 M：数据挖掘（Mining）
# 选择算法（决策树）并在训练集上学习参数。
# 同时在验证集上调整超参数 max_depth。
# ─────────────────────────────────────────────────────────────
print("\n【步骤 M：数据挖掘（建模）】")

best_depth, best_val_acc = 1, 0.0
for depth in range(1, 10):
    clf = DecisionTreeClassifier(max_depth=depth, random_state=42)
    clf.fit(X_train_s, y_train)
    val_acc = accuracy_score(y_val, clf.predict(X_val_s))
    if val_acc > best_val_acc:
        best_val_acc, best_depth = val_acc, depth

print(f"最优 max_depth（按验证集准确率选择）：{best_depth}")
print(f"验证集最高准确率：{best_val_acc:.4f}")

# 以最优超参数重新训练（在 train+val 的并集上）
X_trainval = np.vstack([X_train_s, X_val_s])
y_trainval = np.concatenate([y_train, y_val])
final_clf = DecisionTreeClassifier(max_depth=best_depth, random_state=42)
final_clf.fit(X_trainval, y_trainval)

# ─────────────────────────────────────────────────────────────
# 步骤 E：解释与评估（Evaluation/Interpretation）
# 在封存的测试集上运行一次，报告最终性能。
# 分析决策树规则，评估其业务可解释性。
# ─────────────────────────────────────────────────────────────
print("\n【步骤 E：解释与评估】")

y_pred = final_clf.predict(X_test_s)
test_acc = accuracy_score(y_test, y_pred)

print(f"\n测试集准确率（最终性能，仅报告一次）：{test_acc:.4f}")
print(f"\n分类报告（每类的 Precision / Recall / F1）：")
print(classification_report(y_test, y_pred,
                             target_names=iris.target_names))

print("混淆矩阵（行=真实类别，列=预测类别）：")
print(confusion_matrix(y_test, y_pred))

# 解释：决策树的特征重要性
feat_importance = dict(zip(iris.feature_names, final_clf.feature_importances_))
print("\n特征重要性（基于 Gini 增益）：")
for feat, imp in sorted(feat_importance.items(), key=lambda x: -x[1]):
    bar = "█" * int(imp * 30)
    print(f"  {feat:<30} {imp:.4f}  {bar}")
# 期望：花瓣长度/宽度占主导，与生物学直觉一致

print("\n" + "=" * 60)
print("  KDD 五步流程演示完成")
print(f"  数据路径：原始数据({len(iris.data)}) → 选择({len(X_selected)}) "
      f"→ 预处理({len(X_selected)}) → 训练({len(X_train)}) → 测试({len(X_test)})")
print("=" * 60)
```

{% hint style="tip" %}
💡 **技巧：`Pipeline` 是防止数据泄露的最可靠工具**

本节代码通过手动管理 `fit()` 与 `transform()` 的调用顺序来防止泄露，但在更复杂的流程（如嵌套超参数搜索）中，手动管理极易出错。`sklearn.pipeline.Pipeline` 通过将 `Scaler`、`PCA`、`Classifier` 串联为单一估计器，在 `cross_val_score()` 或 `GridSearchCV()` 内部自动保证：每次折叠（fold）的验证集在 `fit()` 步骤中完全被排除。这是工业代码规范的最低要求——**永远将变换步骤包裹在 Pipeline 中**，而非手动调用 `fit_transform(data_all)`。
{% endhint %}

## 本节小结

本节建立了数据挖掘项目的方法论底座，核心要点如下：

1. **KDD 流程**：将知识发现形式化为五步数据变换复合链 $$\mathcal{D} \xrightarrow{S} \mathcal{D}' \xrightarrow{P} \mathcal{D}'' \xrightarrow{T} \mathcal{D}''' \xrightarrow{M} f \xrightarrow{E} \mathcal{P}$$（定义 1.7），本质上是带有反馈循环的迭代过程，而非线性流水线。
2. **CRISP-DM**：工业界最广泛采用的六阶段项目框架，以业务理解为起点、以部署为终点，并以循环箭头强调迭代改进的持续性（如图 1.3.1）。
3. **三框架比较**：KDD 提供形式化语言；CRISP-DM 提供业务管理规范；SEMMA 为 SAS 工具用户设计——三者互补而非竞争。
4. **数据类型**：结构化（关系表/时序）→ 半结构化（JSON/图）→ 非结构化（文本/图像），挖掘难度依次递增，所需表示学习能力依次提升（如图 1.3.2）。
5. **数据质量五维度**：完整性 $$\rho_c$$（1.11）、唯一性 $$\rho_u$$（1.12）、一致性 $$\rho_k$$（1.13）、准确性、时效性，是 CRISP-DM 数据理解阶段的核心产出。
6. **数据集三分法**：训练/验证/测试集须严格不相交（定义 1.9）；所有变换参数仅在训练集上 `fit()`；测试集封存至最终评估——这是可信性能报告的基本保证。

{% hint style="frontier" %}
🚀 **前沿视野：MLOps 与自动化 CRISP-DM（2022–2025）**

CRISP-DM 的六个阶段中，哪些已经可以被自动化？过去三年的 MLOps 工具链正在逐步回答这个问题：

**阶段二（数据理解）的部分自动化**：数据质量工具（Great Expectations、Soda Core）可自动生成数据质量报告，检测 Schema 漂移和分布变化，将人工 EDA 中的例行工作自动化。

**阶段三（数据准备）的 AI 辅助**：AutoML 系统（AutoGluon、H2O AutoML）内置了自动特征工程、自动编码与自动缺失值处理，可将数据准备时间从"数周"压缩至"数小时"；Text-to-SQL 与 AI 数据助手（如 Julius AI）使非技术人员也能完成基础数据准备。

**阶段四（建模）的自动化**：AutoML 的超参数优化（Optuna、Ray Tune）可在数小时内遍历数百种算法-参数组合，在中小数据集上逼近甚至超越专家调参。

**尚无法充分自动化的环节**：阶段一（业务理解）需要深度的领域知识与利益相关方沟通能力；阶段五（业务评估）中，"模型是否解决了正确的业务问题"依赖人类判断；阶段六（部署）的可靠性工程与合规性审查也需要人类负责。

**结论**：自动化工具使数据科学家从繁琐的机械工作中解放出来，但 CRISP-DM 中需要领域理解、业务判断与工程责任感的环节，仍是人类不可替代的价值核心。
{% endhint %}

{% hint style="ideology" %}
🌏 **课程思政：数据治理与数据主权——CRISP-DM 中的合规责任**

CRISP-DM 流程的每个阶段都与数据合规相关联，这在中国《数据安全法》（2021）与《个人信息保护法》（2021，以下简称"个保法"）施行后尤为重要：

**阶段一（业务理解）**：需明确数据使用的合法性基础。个保法第 13 条规定，处理个人信息须满足以下之一：当事人同意、合同必要、法定义务、公共利益等。在项目立项阶段，数据科学家需确认业务场景满足合法性要求，而非在建模完成后再补充合规文件。

**阶段二（数据理解）**：需评估数据集中是否含有个人信息、敏感个人信息（如生物识别、医疗健康、金融账户）。敏感数据须"单独同意"（个保法第 29 条），并实施严格的访问控制与数据最小化原则。

**阶段三（数据准备）**：数据脱敏（de-identification）和匿名化处理须符合标准（如差分隐私，将在第 13 章介绍）。需注意：即使数据已删除直接标识符，若与其他数据集关联后可重新识别个人，仍属个人信息。

**阶段六（部署）**：自动化决策系统（如信贷、医疗、招聘）须满足个保法第 24 条的"说明理由"义务——当事人有权要求对重要决策进行说明并申请复核。

从数据采集设计到模型部署维护，**每一行代码都可能承载法律责任**。数据科学家的职业伦理，不仅体现在技术能力，更体现在将合规意识嵌入每个 CRISP-DM 阶段的工程实践中。
{% endhint %}
