## 6.7.1 关联规则挖掘工程管线

无论应用场景如何变化，关联规则挖掘的工程实践都可以抽象为一个标准化的管线。

{% block type="definition" title="定义 6.27　关联规则挖掘工程管线" %}
**关联规则挖掘工程管线** $\Pi$ 是一个六阶段流程：

$$
\Pi: \mathcal{D}_{\text{raw}} \xrightarrow{\text{Stage 1}} \mathcal{D} \xrightarrow{\text{Stage 2}} \mathcal{F} \xrightarrow{\text{Stage 3}} \mathcal{R} \xrightarrow{\text{Stage 4}} \mathcal{R}^* \xrightarrow{\text{Stage 5}} \mathcal{V} \xrightarrow{\text{Stage 6}} \mathcal{A} \tag{6.53}
$$

各阶段定义如下：

| 阶段 | 名称 | 输入 → 输出 | 关键操作 |
|------|------|-----------|---------|
| Stage 1 | 数据预处理 | $\mathcal{D}_{\text{raw}} \to \mathcal{D}$ | 清洗、编码、离散化（§6.5） |
| Stage 2 | 频繁项集挖掘 | $\mathcal{D} \to \mathcal{F}$ | Apriori（§6.2）或 FP-Growth（§6.3） |
| Stage 3 | 规则生成 | $\mathcal{F} \to \mathcal{R}$ | 从频繁项集生成满足 minconf 的规则 |
| Stage 4 | 规则过滤 | $\mathcal{R} \to \mathcal{R}^*$ | 兴趣度量筛选（§6.4）、冗余消除（§6.6） |
| Stage 5 | 可视化分析 | $\mathcal{R}^* \to \mathcal{V}$ | 网络图、热力图、散点图（§6.6） |
| Stage 6 | 业务决策 | $\mathcal{V} \to \mathcal{A}$ | 规则解释、可操作性评估（定理 6.13） |

其中 $\mathcal{A}$ 为最终的可操作建议集合。
{% endblock %}

管线的核心思想是**渐进精炼**：每个阶段都在减少候选集的规模（从海量原始数据到少量可操作建议），同时提升信息的质量和可操作性。下面通过三个案例展示管线的实际运作。

## 6.7.2 案例一：零售购物篮分析

### 6.7.2.1 问题背景

某连锁超市拥有 50 万条交易记录，涵盖约 500 种商品。市场营销团队希望通过关联规则挖掘：(1) 发现互补商品对，指导捆绑促销；(2) 识别商品群组，优化货架布局；(3) 建立交叉推荐策略。

### 6.7.2.2 数据预处理（Stage 1）

购物篮数据的预处理主要包括以下步骤：

1. **去除低频商品**：删除支持度低于 0.1% 的商品（在 50 万条记录中出现不到 500 次的商品），避免这些"长尾"商品产生无意义规则。处理后约保留 200 种商品。
2. **概念层次映射**（§6.5）：将商品映射到两层概念层次（如"2% 牛奶"→"牛奶"→"乳制品"），以便在不同粒度上分析。
3. **时间窗口切分**：将一个顾客在同一天的多次交易合并为一条事务（一个购物"篮子"）。

### 6.7.2.3 挖掘与过滤（Stage 2–4）

参数选择是实际应用中的关键决策。对于零售场景，典型参数为：

$$
\sigma_{\min} = 0.01, \quad \delta_{\min} = 0.3, \quad \text{lift}_{\min} = 1.2 \tag{6.54}
$$

这些阈值的含义是：规则至少覆盖 1% 的交易（约 5000 条），置信度至少 30%，且关联强度至少比随机共现高 20%。

在过滤阶段，还需要额外应用以下领域特定的过滤器：

- **排除互斥品**：同一品类不同品牌的商品（如"可口可乐→百事可乐"）的高关联规则通常源于价格促销而非真实偏好。
- **排除配套品**：如"打印机→墨盒"这类已知的配套关系不提供新信息。
- **聚焦高利润品**：将后件限定为高利润商品，优先发现"X→高利润品"的交叉销售机会。

### 6.7.2.4 可视化与决策（Stage 5–6）

经过完整管线处理后，50 万条原始交易 → 约 3000 条频繁项集 → 约 800 条规则 → 过滤后约 120 条非冗余规则 → 最终 15–20 条可操作建议。

## 6.7.3 案例二：医疗诊断中的症状-疾病关联

### 6.7.3.1 问题背景

电子病历（Electronic Medical Records, EMR）包含丰富的症状、诊断和用药信息。通过关联规则挖掘，可以发现症状之间的共现模式，辅助临床决策。

{% block type="definition" title="定义 6.28　症状共现模式" %}
给定电子病历数据库 $\mathcal{D}_{\text{EMR}}$，其中每条记录 $t_i = (S_i, D_i)$，$S_i$ 为患者的症状集合，$D_i$ 为最终诊断集合。**症状-疾病关联规则** 定义为：

$$
r: S_{\text{sub}} \to d, \quad S_{\text{sub}} \subseteq S_i, \quad d \in D_i \tag{6.55}
$$

其中 $S_{\text{sub}}$ 为症状子集（前件），$d$ 为疾病诊断（后件）。

与零售场景不同，医疗场景要求**高置信度优先**——漏诊（假阴性）的代价远高于过度检查（假阳性）。因此参数设置通常为：

$$
\sigma_{\min} = 0.005, \quad \delta_{\min} = 0.7 \tag{6.56}
$$

即允许较低的支持度（覆盖 0.5% 的病例即可），但要求很高的置信度（70% 以上）。
{% endblock %}

医疗场景的关联规则挖掘有以下特殊挑战：

1. **高维稀疏性**：ICD-10 诊断编码有上万种，大多数诊断的出现频率极低。需要利用概念层次（§6.5）将具体诊断映射到更粗的疾病类别。
2. **因果性要求**：零售场景中的关联规则不要求因果关系（"啤酒→薯片"只是共现，不是因果），但医疗场景中"症状→疾病"暗含因果假设，需要额外的领域知识验证。
3. **伦理约束**：挖掘出的规则不能直接用于自动诊断，只能作为辅助参考。规则的解释必须由领域专家审核。

### 6.7.3.2 实现要点

```
典型发现示例：
  {发热, 咳嗽, 胸痛} → 肺炎    (supp=0.012, conf=0.78, lift=5.2)
  {头痛, 恶心, 视觉模糊} → 偏头痛  (supp=0.008, conf=0.82, lift=6.1)
  {关节痛, 晨僵, 疲劳} → 类风湿关节炎 (supp=0.005, conf=0.71, lift=8.3)
```

这些规则不能替代临床诊断，但可以帮助初诊筛查和分诊——当系统检测到某个症状组合的置信度超过阈值时，可以提醒医生关注特定疾病的可能性。

## 6.7.4 案例三：Web 点击流分析与推荐

### 6.7.4.1 问题背景

电商网站的用户浏览记录构成"点击流"（clickstream）数据。与购物篮数据不同，点击流具有**时序性**——页面的浏览顺序包含重要信息。

{% block type="definition" title="定义 6.29　序列关联规则" %}
给定用户会话集合 $\mathcal{S} = \{s_1, s_2, \ldots, s_m\}$，其中 $s_i = \langle p_{i1}, p_{i2}, \ldots, p_{i,l_i} \rangle$ 为页面访问序列。**序列关联规则**定义为：

$$
r: \langle p_a, p_b \rangle \to p_c \tag{6.57}
$$

表示"依次浏览了页面 $p_a$ 和 $p_b$ 的用户倾向于接下来浏览页面 $p_c$"。

其支持度和置信度定义为：

$$
\text{supp}(r) = \frac{|\{s \in \mathcal{S} : \langle p_a, p_b, p_c \rangle \subseteq s\}|}{|\mathcal{S}|}, \quad \text{conf}(r) = \frac{\text{supp}(\langle p_a, p_b, p_c \rangle)}{\text{supp}(\langle p_a, p_b \rangle)} \tag{6.58}
$$

其中 $\langle p_a, p_b, p_c \rangle \subseteq s$ 表示序列 $s$ 包含子序列 $\langle p_a, p_b, p_c \rangle$（不要求连续，但要求保持相对顺序）。
{% endblock %}

序列关联规则与普通关联规则的关键区别在于它保留了顺序信息。例如，"手机配件页→手机壳页→结算页"这条序列规则暗示用户在浏览配件后有较高的购买意向，可以在"手机配件页"上主动推荐"手机壳页"的链接。

### 6.7.4.2 从关联规则到推荐系统

关联规则可以自然地转化为推荐策略：

| 推荐场景 | 使用的规则类型 | 推荐逻辑 |
|---------|-------------|---------|
| "猜你喜欢" | 常规关联规则 | 用户购买了 A，推荐 B（若 A→B 成立） |
| "浏览此商品的人还浏览了" | 序列关联规则 | 用户正在浏览 A，推荐 B（若 ⟨A⟩→B 成立） |
| "经常一起买" | 频繁项集 | 展示 A 的互补品集合 |
| "买了又买" | 跨时间窗关联规则 | 用户上次购买了 A，本次推荐 B |

这种基于关联规则的推荐方法具有**完全可解释性**——每条推荐背后都有明确的统计证据（支持度、置信度、提升度），这在需要解释推荐理由的场景（如金融产品推荐）中尤为重要。

### 6.7.4.3 与协同过滤的比较

基于关联规则的推荐和协同过滤（Collaborative Filtering）都是推荐系统的经典方法。两者的核心区别在于：

| 特征 | 关联规则推荐 | 协同过滤 |
|------|-----------|---------|
| 数据粒度 | 事务级（购物篮） | 用户级（评分矩阵） |
| 个性化程度 | 低（基于全局统计） | 高（基于用户相似度） |
| 可解释性 | 强（规则 + 度量） | 弱（矩阵分解不可解释） |
| 冷启动 | 仅需少量事务数据 | 需要足够的用户评分 |
| 计算复杂度 | 挖掘阶段高，推荐阶段低 | 模型训练高 |
| 适用场景 | 高频低价品、超市 | 低频高价品、影视/音乐 |

## 6.7.5 关联规则系统的评估

在实际部署前，需要评估关联规则挖掘系统的有效性。与分类模型不同，关联规则没有单一的"准确率"指标，需要从多个维度综合评估。

{% block type="theorem" title="定理 6.14　关联规则系统的评估框架" %}
设关联规则挖掘系统 $\Pi$ 在数据集 $\mathcal{D}$ 上产生规则集 $\mathcal{R}$，其有效性可从以下四个维度评估：

**维度一：统计有效性**——规则是否反映真实的统计关联（而非采样偏差或多重检验伪发现）。

$$
\text{FDR}(\mathcal{R}) = \frac{|\{r \in \mathcal{R} : r \text{ 是伪发现}\}|}{|\mathcal{R}|} \tag{6.59}
$$

当搜索空间很大时（如 $|\mathcal{I}| > 100$），需要应用 Bonferroni 校正或 BH 过程控制 FDR。

**维度二：稳定性**——规则在不同数据子集上是否稳定。定义 Bootstrap 稳定性：

$$
\text{Stability}(r) = \frac{1}{B} \sum_{b=1}^{B} \mathbb{1}[r \in \mathcal{R}_b] \tag{6.60}
$$

其中 $\mathcal{R}_b$ 为第 $b$ 次 Bootstrap 抽样得到的规则集。$\text{Stability}(r) > 0.8$ 表示规则稳定。

**维度三：可操作性**——规则是否能转化为有效的业务动作（定理 6.13）。

**维度四：业务效果**——规则驱动的业务动作是否产生可衡量的效果。典型指标包括：

- 捆绑促销的销售提升率
- 推荐系统的点击率（CTR）和转化率
- 货架优化后的品类销售增长

**证明**（必要性）：任何评估框架至少需要覆盖这四个维度，因为：
- 缺少维度一 → 可能基于伪发现做决策（§6.4 中 $\chi^2$ 检验的意义）；
- 缺少维度二 → 规则可能仅在特定时间段有效（过拟合到季节性波动）；
- 缺少维度三 → 发现大量"正确但无用"的规则浪费分析资源；
- 缺少维度四 → 无法衡量 ROI，挖掘系统的价值无法量化。$\blacksquare$
{% endblock %}

在实际项目中，维度四（业务效果）通常通过 **A/B 测试** 来评估：将用户随机分为实验组（应用关联规则驱动的策略）和对照组（使用原始策略），比较两组的关键业务指标。

## 6.7.6 Python 实现：完整端到端管线

下面实现一个零售购物篮分析的完整管线，涵盖数据生成、挖掘、过滤、可视化和评估。

{% block type="code" title="代码 6.10　零售购物篮完整分析管线" %}
```python
import numpy as np
from itertools import combinations
from collections import defaultdict
np.random.seed(42)
# ── 1. Stage 1：数据预处理 ─────────────────────────────────
items = ['面包', '牛奶', '鸡蛋', '啤酒', '薯片', '可乐',
         '酸奶', '水果', '蔬菜', '肉类', '调料', '饮料']
profit_margin = {'面包': 0.15, '牛奶': 0.10, '鸡蛋': 0.12,
                 '啤酒': 0.35, '薯片': 0.45, '可乐': 0.40,
                 '酸奶': 0.20, '水果': 0.25, '蔬菜': 0.20,
                 '肉类': 0.18, '调料': 0.50, '饮料': 0.38}
n_trans = 5000
transactions = []
for _ in range(n_trans):
    basket = set()
    # 注入关联模式
    if np.random.random() < 0.30:
        basket.update(['面包', '牛奶'])
        if np.random.random() < 0.45: basket.add('鸡蛋')
    if np.random.random() < 0.22:
        basket.update(['啤酒', '薯片'])
        if np.random.random() < 0.35: basket.add('可乐')
    if np.random.random() < 0.15:
        basket.update(['酸奶', '水果'])
    if np.random.random() < 0.10:
        basket.update(['肉类', '蔬菜', '调料'])
    for item in items:
        if item not in basket and np.random.random() < 0.08:
            basket.add(item)
    if basket:
        transactions.append(basket)
n = len(transactions)
# ── 2. Stage 2：频繁项集挖掘 ───────────────────────────────
min_support, min_confidence, min_lift = 0.01, 0.25, 1.0
min_count = int(min_support * n)
item_counts = defaultdict(int)
for t in transactions:
    for item in t:
        item_counts[item] += 1
freq_items = {item: cnt for item, cnt in item_counts.items()
              if cnt >= min_count}
freq_pairs = {}
for a, b in combinations(freq_items.keys(), 2):
    cnt = sum(1 for t in transactions if a in t and b in t)
    if cnt >= min_count:
        freq_pairs[frozenset([a, b])] = cnt
# ── 3. Stage 3：规则生成 ───────────────────────────────────
rules = []
for pair, pair_cnt in freq_pairs.items():
    a, b = list(pair)
    pAB, pA, pB = pair_cnt/n, freq_items[a]/n, freq_items[b]/n
    conf_ab, conf_ba = pAB/pA, pAB/pB
    lift = pAB / (pA * pB)
    kulc = 0.5 * (conf_ab + conf_ba)
    ir = abs(pA - pB) / (pA + pB - pAB) if (pA + pB - pAB) > 0 else 0
    for ant, con, conf in [(a, b, conf_ab), (b, a, conf_ba)]:
        if conf >= min_confidence and lift >= min_lift:
            rules.append({'ant': ant, 'con': con, 'supp': pAB,
                          'conf': conf, 'lift': lift,
                          'kulc': kulc, 'ir': ir})
# ── 4. Stage 4：规则过滤与冗余消除 ─────────────────────────
filtered = [r for r in rules if r['lift'] > 1.1]
by_con = defaultdict(list)
for r in filtered:
    by_con[r['con']].append(r)
clean_rules = []
for con, group in by_con.items():
    group.sort(key=lambda r: (-r['conf'], r['supp']))
    clean_rules.extend(group[:3])  # 每个后件保留 Top-3
for r in clean_rules:
    r['action_score'] = r['lift'] * r['supp']
clean_rules.sort(key=lambda r: -r['action_score'])
# ── 5. Stage 5–6：商业建议与预期价值 ──────────────────────
for i, r in enumerate(clean_rules[:10], 1):
    con_profit = profit_margin.get(r['con'], 0.2)
    ev = r['supp'] * r['conf'] * con_profit * n  # 预期价值
    if r['lift'] > 2.0 and r['supp'] > 0.05:
        advice = "捆绑促销（高关联+高覆盖）"
    elif r['lift'] > 2.0:
        advice = "交叉推荐（高关联）"
    elif r['supp'] > 0.10:
        advice = "优化货架（高频共现）"
    else:
        advice = "监控关注"
    print(f"#{i} {r['ant']}→{r['con']}  supp={r['supp']:.3f} "
          f"conf={r['conf']:.3f} lift={r['lift']:.2f} "
          f"EV=¥{ev:.0f}  → {advice}")
# ── 6. 评估：Bootstrap 稳定性（定理 6.14）─────────────────
B = 20
top5 = [(r['ant'], r['con']) for r in clean_rules[:5]]
stability = {rule: 0 for rule in top5}
for _ in range(B):
    idx = np.random.choice(n, n, replace=True)
    boot = [transactions[i] for i in idx]
    for ant, con in top5:
        pAB = sum(1 for t in boot if ant in t and con in t) / n
        pA = sum(1 for t in boot if ant in t) / n
        if pA > 0:
            pB = sum(1 for t in boot if con in t) / n
            conf = pAB / pA
            lift = pAB / (pA * pB) if pA * pB > 0 else 0
            if conf >= min_confidence and lift >= min_lift:
                stability[(ant, con)] += 1
for rule, cnt in stability.items():
    stab = cnt / B
    tag = "✓ 稳定" if stab > 0.8 else "⚠ 不稳定"
    print(f"  {rule[0]}→{rule[1]}: Stability={stab:.2f} {tag}")
```
{% endblock %}

上述代码完整实现了管线的六个阶段。值得注意的是 Stage 6 中的"预期价值"计算——它将统计度量（支持度、置信度）与业务指标（利润率）结合，这正是将"统计有趣"转化为"商业有用"的关键步骤。

## 6.7.7 关联规则挖掘的局限性与未来方向

尽管关联规则挖掘在过去三十年取得了巨大成功，但它也存在固有的局限性。理解这些局限对于正确应用这一技术至关重要。

### 6.7.7.1 主要局限性

1. **关联不等于因果**：关联规则发现的是统计共现模式，不能直接推断因果关系。例如，"雨伞→雨衣"的高关联来自共同的原因（下雨），而非"买雨伞导致买雨衣"。需要因果推断方法（如工具变量、随机实验）来建立因果关系。

2. **组合爆炸**：即使有 Apriori 剪枝和 FP-Growth 优化，当项数超过 $10^4$ 时（如基因表达数据），频繁项集挖掘仍然面临巨大的计算挑战。

3. **阈值敏感性**：minsup 和 minconf 的微小变化可能导致规则集的剧烈变化。没有普适的阈值选择准则——不同数据集和应用场景需要不同的阈值。

4. **静态分析**：标准关联规则假设数据是静态的。在实际中，消费者行为随时间变化（季节性、趋势、事件驱动），需要增量挖掘或滑动窗口方法。

5. **忽略负关联**：标准方法专注于正关联（$\text{lift} > 1$），忽略了负关联（$\text{lift} < 1$）。但"买了 A 的人不太可能买 B"的信息在某些场景下同样有价值。

### 6.7.7.2 未来方向

| 方向 | 核心思想 | 代表工作 |
|------|---------|---------|
| 增量挖掘 | 数据更新时只需更新受影响的项集 | FUP/FUP2 算法 |
| 流数据关联 | 在数据流上维护近似频繁项集 | Lossy Counting |
| 隐私保护 | 在保护个体隐私的前提下挖掘全局模式 | 差分隐私关联挖掘 |
| 图关联规则 | 在图数据上发现频繁子图模式 | gSpan, FSG |
| 因果发现 | 从关联规则中推断因果关系 | PC 算法 + 关联分析 |
| 深度学习融合 | 用神经网络学习项的隐表示辅助挖掘 | Neural Association Rules |
