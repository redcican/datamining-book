# 6.2 Apriori 算法

§6.1 建立了关联规则挖掘的数学框架，并指出核心计算瓶颈在于**频繁项集发现**——从 $2^m$ 个候选中找出所有频繁项集。暴力枚举不可行，而 Apriori 性质（定理 6.1）提供了强大的剪枝工具。

1994 年，Agrawal 和 Srikant 提出了 **Apriori 算法**——第一个系统利用 Apriori 性质进行逐层搜索的频繁项集挖掘算法。Apriori 的核心思想简洁而深刻：**从频繁 1-项集出发，每一轮利用上一轮的频繁 $k$-项集生成候选 $(k+1)$-项集，通过剪枝和支持度计数得到频繁 $(k+1)$-项集，直到无法继续为止**。尽管后来的 FP-Growth（§6.3）在性能上超越了 Apriori，但 Apriori 的逐层剪枝思想至今仍是关联规则领域最重要的算法范式。

## 6.2.1 从 Apriori 性质到逐层搜索

### 6.2.1.1 逐层搜索的直觉

Apriori 性质告诉我们：**频繁项集的所有子集都频繁，非频繁项集的所有超集都非频繁**。这意味着我们可以按项集大小从小到大搜索——先找频繁 1-项集 $\mathcal{F}_1$，再用 $\mathcal{F}_1$ 构造候选 2-项集 $C_2$，验证后得到 $\mathcal{F}_2$，再用 $\mathcal{F}_2$ 构造 $C_3$……每一层只保留频繁项集，非频繁的分支被彻底剪除。

这个过程可以用一个关键不变量来理解：

$$
\mathcal{F}_{k+1} \subseteq C_{k+1} \subseteq \{X \cup Y : X, Y \in \mathcal{F}_k, |X \cup Y| = k+1\} \tag{6.12}
$$

即频繁 $(k+1)$-项集一定是从频繁 $k$-项集中"生长"出来的。

### 6.2.1.2 算法的宏观流程

```text
算法：Apriori(D, minsup)
──────────────────────────────────────────
输入：事务数据库 D，最小支持度阈值 minsup
输出：所有频繁项集 F

1. 扫描 D，计算每个 1-项集的支持度
2. F_1 ← {频繁 1-项集}
3. k ← 1
4. while F_k ≠ ∅:
5.     C_{k+1} ← 候选生成(F_k)        // 连接步 + 剪枝步
6.     扫描 D，计算 C_{k+1} 中每个候选的支持度
7.     F_{k+1} ← {c ∈ C_{k+1} : supp(c) ≥ minsup}
8.     k ← k + 1
9. return F = F_1 ∪ F_2 ∪ ... ∪ F_k
```

注意，每次迭代需要**完整扫描一遍数据库**——这是 Apriori 的主要 I/O 代价。若最长频繁项集为 $k$-项集，则需要 $k+1$ 次数据库扫描（包括初始的 1-项集计数）。

## 6.2.2 候选项集生成：连接步与剪枝步

Apriori 算法的核心操作是**候选项集生成**——如何从 $\mathcal{F}_k$ 高效地构造 $C_{k+1}$，使得 $C_{k+1}$ 包含所有可能的频繁 $(k+1)$-项集，同时尽量减少非频繁候选的数量。

{% block type="definition" title="定义 6.7　Apriori 候选生成（Candidate Generation）" %}
给定频繁 $k$-项集的集合 $\mathcal{F}_k$（其中每个项集的元素按固定的全序排列），候选 $(k+1)$-项集的生成分两步：

**连接步**（Join Step）：对 $\mathcal{F}_k$ 中任意两个项集 $L_1 = \{l_1, \ldots, l_{k-1}, l_k\}$ 和 $L_2 = \{l_1, \ldots, l_{k-1}, l_k'\}$，若它们的**前 $k-1$ 个元素完全相同**，且 $l_k < l_k'$（按字典序），则生成候选：

$$
c = L_1 \cup L_2 = \{l_1, \ldots, l_{k-1}, l_k, l_k'\} \tag{6.13}
$$

**剪枝步**（Prune Step）：对连接步产生的每个候选 $c$，检查其所有 $k$ 个子集是否都在 $\mathcal{F}_k$ 中。若存在某个 $k$-子集不在 $\mathcal{F}_k$ 中，则根据 Apriori 性质，$c$ 不可能是频繁的，将其从 $C_{k+1}$ 中移除：

$$
C_{k+1} = \{c : c \text{ 由连接步生成} \wedge \forall s \subset c, |s|=k \implies s \in \mathcal{F}_k\} \tag{6.14}
$$
{% endblock %}

为了使连接步的含义更加具体，我们来看一个完整的例子。

### 6.2.2.1 候选生成的详细示例

设 $\mathcal{F}_3 = \{\{A,B,C\}, \{A,B,D\}, \{A,C,D\}, \{A,C,E\}, \{B,C,D\}\}$（项按字典序排列）。

**连接步**：找前 2 个元素相同的配对：

| 配对 | 前缀 | 候选 |
|------|------|------|
| $\{A,B,C\} \bowtie \{A,B,D\}$ | $\{A,B\}$ | $\{A,B,C,D\}$ |
| $\{A,C,D\} \bowtie \{A,C,E\}$ | $\{A,C\}$ | $\{A,C,D,E\}$ |

**剪枝步**：

- 候选 $\{A,B,C,D\}$：检查 4 个 3-子集——$\{A,B,C\} \in \mathcal{F}_3$ ✓，$\{A,B,D\} \in \mathcal{F}_3$ ✓，$\{A,C,D\} \in \mathcal{F}_3$ ✓，$\{B,C,D\} \in \mathcal{F}_3$ ✓——**保留**。
- 候选 $\{A,C,D,E\}$：检查 3-子集 $\{A,D,E\}$——$\{A,D,E\} \notin \mathcal{F}_3$ ✗——**剪掉**。

因此 $C_4 = \{\{A,B,C,D\}\}$。

## 6.2.3 候选生成的完备性与正确性

一个自然的问题是：定义 6.7 的连接-剪枝过程是否会**遗漏**真正的频繁项集？以下定理给出了否定的回答。

{% block type="theorem" title="定理 6.3　Apriori 候选生成的完备性" %}
设 $\mathcal{F}_k$ 是所有频繁 $k$-项集的完整集合。则定义 6.7 的连接步生成的候选集合 $C_{k+1}$（**剪枝前**）包含所有频繁 $(k+1)$-项集。即：

$$
\mathcal{F}_{k+1} \subseteq C_{k+1}^{\text{join}} \tag{6.15}
$$

**证明**：

设 $F = \{i_1, i_2, \ldots, i_{k+1}\}$（$i_1 < i_2 < \cdots < i_{k+1}$）是一个频繁 $(k+1)$-项集。由 Apriori 性质（定理 6.1），$F$ 的所有 $k$-子集都频繁，特别是：

$$
L_1 = \{i_1, \ldots, i_{k-1}, i_k\} \in \mathcal{F}_k, \quad L_2 = \{i_1, \ldots, i_{k-1}, i_{k+1}\} \in \mathcal{F}_k
$$

$L_1$ 和 $L_2$ 的前 $k-1$ 个元素完全相同（均为 $i_1, \ldots, i_{k-1}$），且 $i_k < i_{k+1}$。因此 $L_1$ 和 $L_2$ 满足连接步的条件，$F = L_1 \cup L_2$ 会被生成。$\square$

由此推论：**剪枝步**只会移除非频繁候选，不会误删频繁项集（因为频繁项集的所有子集都在 $\mathcal{F}_k$ 中），所以 Apriori 算法能**找到所有频繁项集**。
{% endblock %}

完备性保证了算法不遗漏，但候选集合可能包含大量非频繁项集——这正是剪枝步的价值所在。剪枝步利用 Apriori 性质提前排除不可能频繁的候选，减少了后续支持度计数的代价。

## 6.2.4 支持度计数

候选生成后，Apriori 需要**扫描整个数据库**来计算每个候选的支持度。这是算法最耗时的步骤。

### 6.2.4.1 朴素方法

对每个事务 $T$ 和每个候选 $c \in C_{k+1}$，检查 $c \subseteq T$。若 $|C_{k+1}| = p$，事务数为 $n$，则单次扫描的代价为 $O(n \cdot p \cdot k)$。

### 6.2.4.2 哈希树优化

为避免逐一检查每个候选，Agrawal 和 Srikant 在原始论文中提出了**哈希树**（hash tree）数据结构：

1. 将所有候选插入一棵哈希树（叶节点存储候选项集）；
2. 对每个事务 $T$，沿哈希树遍历，快速找到 $T$ 包含的所有候选；
3. 平均情况下将支持度计数加速至 $O(n \cdot \bar{l}^k / k!)$（$\bar{l}$ 为平均事务长度）。

---

## 6.2.5 Apriori 算法的完整伪代码

综合连接步、剪枝步和支持度计数，Apriori 的完整流程如下：

```text
算法：Apriori(D, minsup)
──────────────────────────────────────────
输入：事务数据库 D = {T_1,...,T_n}，最小支持度阈值 minsup
输出：所有频繁项集 F 及其支持度

1.  /* 第一次扫描：频繁 1-项集 */
2.  C_1 ← 所有 1-项集
3.  扫描 D，计算 C_1 中每个项的支持度
4.  F_1 ← {c ∈ C_1 : supp(c) ≥ minsup}
5.  k ← 1
6.
7.  while F_k ≠ ∅:
8.      /* 候选生成 */
9.      C_{k+1} ← ∅
10.     for 每对 (L_1, L_2) ∈ F_k × F_k，L_1[1..k-1] = L_2[1..k-1]，L_1[k] < L_2[k]:
11.         c ← L_1 ∪ {L_2[k]}
12.         /* 剪枝步 */
13.         if c 的所有 k-子集均在 F_k 中:
14.             C_{k+1} ← C_{k+1} ∪ {c}
15.
16.     /* 支持度计数（第 k+1 次扫描） */
17.     for 每个事务 T ∈ D:
18.         for 每个候选 c ∈ C_{k+1}:
19.             if c ⊆ T: σ(c) ← σ(c) + 1
20.
21.     F_{k+1} ← {c ∈ C_{k+1} : σ(c)/n ≥ minsup}
22.     k ← k + 1
23.
24. return F = F_1 ∪ F_2 ∪ ... ∪ F_k
```

## 6.2.6 复杂度分析

Apriori 的性能取决于数据特征和 minsup 阈值。以下定理给出了关键复杂度指标。

{% block type="theorem" title="定理 6.4　Apriori 算法的复杂度" %}
设事务数据库 $\mathcal{D}$ 有 $n$ 个事务，项集域大小为 $m$，最长频繁项集为 $k_{\max}$-项集。

**I/O 复杂度**（数据库扫描次数）：

$$
S = k_{\max} + 1 \tag{6.16}
$$

每轮迭代需完整扫描一遍数据库。

**候选生成的时间复杂度**：连接步需要 $O(|\mathcal{F}_k|^2)$ 次比较，剪枝步对每个候选需检查 $O(k)$ 个子集是否在 $\mathcal{F}_k$ 中（使用哈希表查找为 $O(k)$）。因此单轮候选生成为：

$$
T_{\text{gen}}(k) = O(|\mathcal{F}_k|^2 \cdot k) \tag{6.17}
$$

**支持度计数的时间复杂度**：使用朴素方法为 $O(n \cdot |C_{k+1}| \cdot k)$，使用哈希树可降低到约 $O(n \cdot \bar{l}^k / k!)$。

**空间复杂度**：需要同时存储 $\mathcal{F}_k$ 和 $C_{k+1}$，为 $O(\max_k |C_k| \cdot k)$。
{% endblock %}

上述复杂度揭示了 Apriori 的两个主要瓶颈：

1. **多次数据库扫描**：$k_{\max}+1$ 次全扫描在磁盘数据库上代价高昂——这正是 FP-Growth（§6.3）仅需两次扫描的核心优势所在；
2. **候选集爆炸**：当 minsup 过低时，$|\mathcal{F}_k|$ 可能非常大，导致 $|C_{k+1}|$ 呈二次方增长。

---

## 6.2.7 Apriori 的优化策略

为缓解上述瓶颈，文献中提出了多种优化策略。

### 6.2.7.1 事务压缩（Transaction Reduction）

若一个事务不包含任何频繁 $k$-项集，则它也不可能包含任何频繁 $(k+1)$-项集（因为 $(k+1)$-项集的所有 $k$-子集必须频繁）。因此可以在每轮迭代中**移除**不再贡献的事务，减少后续扫描的数据量。

### 6.2.7.2 分区法（Partition）

将数据库 $\mathcal{D}$ 分为 $p$ 个不重叠的分区 $\mathcal{D}_1, \ldots, \mathcal{D}_p$。算法分两阶段进行：

**阶段 1**：在每个分区 $\mathcal{D}_i$ 上独立运行 Apriori，得到**局部频繁项集** $\mathcal{F}_i$。一个在全局频繁的项集**至少在某个分区中是频繁的**，因此全局候选集为 $C = \bigcup_i \mathcal{F}_i$。

**阶段 2**：扫描整个数据库，计算 $C$ 中每个候选的全局支持度，筛选出真正的频繁项集。

分区法将数据库扫描减少到**恰好 2 次**（阶段 1 各分区各一次可合并为一次全扫描，加上阶段 2 一次），且每个分区可以放入内存，避免了多次磁盘 I/O。

### 6.2.7.3 采样法（Sampling）

从 $\mathcal{D}$ 中**随机采样**一个子集 $\mathcal{S}$（如 1%–10%），在 $\mathcal{S}$ 上用较低的 minsup 运行 Apriori，得到候选集 $C$。然后在完整数据库上验证 $C$ 的支持度。

采样法的优势是速度极快（样本小），风险是可能遗漏全局频繁但在样本中恰好不频繁的项集（**假阴性**）。使用稍低的 minsup 阈值可降低这一风险。

### 6.2.7.4 优化策略对比

| 策略 | 数据库扫描次数 | 核心思想 | 风险 |
|------|-------------|---------|------|
| 原始 Apriori | $k_{\max}+1$ | 逐层搜索 | I/O 代价高 |
| 事务压缩 | $k_{\max}+1$（但数据量递减） | 删除无用事务 | 无精度损失 |
| 分区法 | 2 | 局部频繁→全局验证 | 无精度损失 |
| 采样法 | 1–2 | 子集挖掘→全量验证 | 可能有假阴性 |

## 6.2.8 从频繁项集到关联规则

Apriori 找到所有频繁项集后，关联规则的生成是相对直接的。对每个频繁项集 $L$（$|L| \geq 2$），枚举所有非空真子集 $X \subset L$，计算 $\text{conf}(X \Rightarrow L \setminus X) = \text{supp}(L) / \text{supp}(X)$，保留置信度 $\geq \text{minconf}$ 的规则。

这个过程也有类似 Apriori 的剪枝性质：**置信度的反单调性**。

{% block type="theorem" title="定理 6.5　关联规则置信度的反单调性" %}
对频繁项集 $L$，若规则 $X \Rightarrow L \setminus X$ 不满足最小置信度，则对任意 $X' \subset X$（$X' \neq \emptyset$），规则 $X' \Rightarrow L \setminus X'$ 也不满足最小置信度：

$$
\text{conf}(X \Rightarrow L \setminus X) < \text{minconf} \implies \text{conf}(X' \Rightarrow L \setminus X') < \text{minconf} \tag{6.18}
$$

**证明**：

由 $X' \subset X$，Apriori 性质给出 $\text{supp}(X') \geq \text{supp}(X)$。因此：

$$
\text{conf}(X' \Rightarrow L \setminus X') = \frac{\text{supp}(L)}{\text{supp}(X')} \leq \frac{\text{supp}(L)}{\text{supp}(X)} = \text{conf}(X \Rightarrow L \setminus X) < \text{minconf} \tag{6.19}
$$

$\square$
{% endblock %}

这意味着：如果以 $\{A,B\}$ 为前件的规则 $\{A,B\} \Rightarrow \{C,D\}$ 不满足置信度，那么前件更小的规则（如 $\{A\} \Rightarrow \{B,C,D\}$）也一定不满足——可以直接剪掉，无需计算。

## 6.2.9 Python 从零实现

```python
import numpy as np
from itertools import combinations

def apriori(transactions, minsup=0.3, minconf=0.6):
    """Apriori 算法从零实现"""
    n = len(transactions)
    min_count = minsup * n
    # ── 第 1 步：频繁 1-项集 ──────────────────────────────
    item_counts = {}
    for t in transactions:
        for item in t:
            item_counts[item] = item_counts.get(item, 0) + 1
    F_1 = {frozenset({item}): count
           for item, count in item_counts.items() if count >= min_count}
    all_frequent = dict(F_1)
    F_k = F_1
    k = 1
    # ── 逐层迭代 ─────────────────────────────────────────
    while F_k:
        k += 1
        # 候选生成：连接步
        items_prev = list(F_k.keys())
        candidates = set()
        for i in range(len(items_prev)):
            for j in range(i + 1, len(items_prev)):
                union = items_prev[i] | items_prev[j]
                if len(union) == k:
                    candidates.add(union)
        # 剪枝步
        C_k = set()
        for c in candidates:
            # 检查所有 (k-1)-子集是否在 F_{k-1} 中
            all_sub_freq = True
            for item in c:
                sub = c - frozenset({item})
                if sub not in F_k:
                    all_sub_freq = False
                    break
            if all_sub_freq:
                C_k.add(c)
        # 支持度计数
        candidate_counts = {c: 0 for c in C_k}
        for t in transactions:
            t_set = frozenset(t)
            for c in C_k:
                if c.issubset(t_set):
                    candidate_counts[c] += 1
        F_k = {c: count for c, count in candidate_counts.items()
               if count >= min_count}
        all_frequent.update(F_k)
    # ── 关联规则生成 ──────────────────────────────────────
    rules = []
    for itemset, count in all_frequent.items():
        sup = count / n
        if len(itemset) < 2:
            continue
        items = list(itemset)
        for r in range(1, len(items)):
            for ante_items in combinations(items, r):
                ante = frozenset(ante_items)
                cons = itemset - ante
                conf = count / all_frequent.get(ante, 1)
                if conf >= minconf:
                    lift = (count / n) / ((all_frequent[ante] / n)
                           * (all_frequent.get(cons, 1) / n))
                    rules.append({
                        'antecedent': ante,
                        'consequent': cons,
                        'support': sup,
                        'confidence': conf,
                        'lift': lift,
                    })
    return all_frequent, rules, n

# ── 示例 ──────────────────────────────────────────────────
transactions = [
    {'面包', '牛奶', '鸡蛋'},
    {'面包', '牛奶', '啤酒', '尿布'},
    {'牛奶', '鸡蛋', '啤酒'},
    {'面包', '牛奶', '鸡蛋', '啤酒'},
    {'面包', '鸡蛋'},
    {'面包', '牛奶', '啤酒'},
    {'牛奶', '鸡蛋'},
    {'面包', '牛奶', '鸡蛋', '啤酒'},
]
frequent, rules, n = apriori(transactions, minsup=0.3, minconf=0.6)

print("频繁项集：")
for itemset, count in sorted(frequent.items(), key=lambda x: (-len(x[0]), -x[1])):
    print(f"  {set(itemset)}: supp={count/n:.3f} (σ={count})")

print(f"\n关联规则（minconf≥0.6）：")
for r in sorted(rules, key=lambda x: -x['lift'])[:10]:
    print(f"  {set(r['antecedent'])} → {set(r['consequent'])}: "
          f"supp={r['support']:.3f}, conf={r['confidence']:.3f}, lift={r['lift']:.2f}")
```

## 案例 6.2：Apriori 算法在购物篮数据上的完整流程

**问题背景**：使用合成的超市购物篮数据（1500 笔交易，12 种商品），从头运行 Apriori 算法，分析不同 minsup 阈值下的频繁项集数量、运行时间和关联规则质量。

```python
import numpy as np
import time
from itertools import combinations

np.random.seed(42)

# ── 1. 生成合成购物篮数据 ─────────────────────────────────
items = ['面包', '牛奶', '鸡蛋', '啤酒', '薯片', '可乐',
         '酸奶', '水果', '蔬菜', '肉类', '饮用水', '零食']
patterns = {
    '早餐': (['面包', '牛奶', '鸡蛋'], 0.40),
    '零食': (['啤酒', '薯片', '可乐'], 0.25),
    '健康': (['水果', '蔬菜', '酸奶'], 0.20),
    '主食': (['肉类', '蔬菜', '鸡蛋'], 0.30),
}

n_trans = 1500
transactions = []
for _ in range(n_trans):
    basket = set()
    for name, (group, prob) in patterns.items():
        if np.random.random() < prob:
            for item in group:
                if np.random.random() < 0.80:
                    basket.add(item)
    n_rand = np.random.poisson(1.5)
    if n_rand > 0:
        basket.update(np.random.choice(items, size=min(n_rand, 4), replace=False))
    if len(basket) > 0:
        transactions.append(basket)

print(f"事务数：{len(transactions)}")
print(f"平均事务长度：{np.mean([len(t) for t in transactions]):.1f}")

# ── 2. 不同 minsup 下的 Apriori ──────────────────────────
minsup_values = [0.30, 0.20, 0.15, 0.10, 0.05]
for ms in minsup_values:
    start = time.time()
    frequent, rules, n = apriori(transactions, minsup=ms, minconf=0.5)
    elapsed = time.time() - start
    n_freq = len(frequent)
    n_rules = len(rules)
    max_k = max(len(fs) for fs in frequent.keys())
    print(f"minsup={ms:.2f}: 频繁项集={n_freq:4d}, 规则={n_rules:4d}, "
          f"最长={max_k}-项集, 耗时={elapsed:.3f}s")

# ── 3. 最有意义的规则（minsup=0.10, minconf=0.5）────────
frequent, rules, n = apriori(transactions, minsup=0.10, minconf=0.5)
print(f"\n前 15 条规则（按提升度降序）：")
for r in sorted(rules, key=lambda x: -x['lift'])[:15]:
    ante = ', '.join(sorted(r['antecedent']))
    cons = ', '.join(sorted(r['consequent']))
    print(f"  {ante} → {cons}: supp={r['support']:.3f}, "
          f"conf={r['confidence']:.3f}, lift={r['lift']:.2f}")

# ── 4. 与 mlxtend 对比验证（如已安装）────────────────────
try:
    import pandas as pd
    from mlxtend.preprocessing import TransactionEncoder
    from mlxtend.frequent_patterns import apriori as mlx_apriori
    from mlxtend.frequent_patterns import association_rules

    te = TransactionEncoder()
    te_ary = te.fit(transactions).transform(transactions)
    df = pd.DataFrame(te_ary, columns=te.columns_)
    freq_mlx = mlx_apriori(df, min_support=0.10, use_colnames=True)
    print(f"\nmlxtend 频繁项集数：{len(freq_mlx)}")
    print(f"自实现频繁项集数：{len(frequent)}")
    print(f"结果一致：{len(freq_mlx) == len(frequent)}")
except ImportError:
    print("\n（mlxtend 未安装，跳过对比验证）")
```

**关键观察**：

1. **minsup 的敏感性**：minsup 从 0.30 降到 0.05 时，频繁项集数量从数十个增长到数千个，运行时间增长 10–100 倍——minsup 是控制算法性能的最关键参数；
2. **规则质量**：高提升度规则集中在人为设计的购物模式上（早餐组：面包↔牛奶↔鸡蛋；零食组：啤酒↔薯片↔可乐），验证了 Apriori 能正确恢复内置模式；
3. **高置信度 ≠ 高价值**：某些规则的置信度很高但提升度接近 1（如"蔬菜→X"——蔬菜出现频率高，几乎与所有商品"共现"），仅凭置信度会被误判为有意义的关联；
4. **逐层收敛**：随着 $k$ 增大，频繁项集数量迅速减少，通常 $k_{\max} \leq 4$——购物篮中超过 4 个商品的频繁组合很少。

## 本节小结

| 关键词 | 核心概念 |
|--------|---------|
| 逐层搜索 | $\mathcal{F}_1 \to C_2 \to \mathcal{F}_2 \to C_3 \to \cdots$，每层一次数据库扫描 |
| 连接步（定义 6.7） | 两个频繁 $k$-项集共享前 $k-1$ 项时合并为 $(k+1)$-项集 |
| 剪枝步（定义 6.7） | 候选的所有 $k$-子集必须都在 $\mathcal{F}_k$ 中，否则剪除 |
| 完备性（定理 6.3） | 连接步不会遗漏任何频繁项集 |
| 复杂度（定理 6.4） | $k_{\max}+1$ 次数据库扫描，候选生成 $O(\|\mathcal{F}_k\|^2 \cdot k)$ |
| 置信度反单调性（定理 6.5） | 前件更小的规则置信度更低，可剪枝 |
| 事务压缩 | 删除不再贡献的事务，减少后续扫描量 |
| 分区法 | 两次扫描：局部频繁 → 全局验证 |
| 采样法 | 子集挖掘 → 全量验证，可能有假阴性 |
