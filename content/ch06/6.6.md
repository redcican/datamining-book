# 6.6 关联规则可视化与解释

§6.1–6.5 系统讨论了关联规则挖掘的算法与度量。然而，一次典型的关联规则挖掘往往会产生**数百甚至数千条规则**——面对如此庞大的规则集合，分析师如何从中快速识别有价值的模式？如何将统计意义上的关联转化为可操作的商业洞察？

答案在于**可视化与解释**。好的可视化不仅能帮助人类在海量规则中"看见"模式，还能揭示规则之间的结构关系（如哪些项是"枢纽"项、规则之间是否存在冗余）。同时，形式化的规则解释框架能帮助区分"统计有趣"与"商业有用"的规则。

本节从三个维度展开：**图结构可视化**（将规则映射为网络）、**度量空间可视化**（在支持度-置信度-提升度空间中展示规则分布）和**规则解释**（冗余消除与可操作性评估）。

## 6.6.1 为什么需要可视化

假设一次 FP-Growth 挖掘在超市数据上产生了 500 条规则。分析师面临的挑战不是"如何找到规则"，而是"如何理解规则"。列表形式的规则集合存在三个根本问题：

1. **信息过载**：人类无法有效处理超过 20–30 条规则的列表，500 条规则意味着大量有价值的模式被"淹没"。
2. **结构不可见**：列表形式无法展示规则之间的关系——哪些项是连接多条规则的"枢纽"？哪些规则形成了有趣的链式模式（A→B, B→C）？
3. **多维度量难以同时对比**：每条规则至少有支持度、置信度、提升度三个维度，表格形式只能按其中一个维度排序，无法同时展示多维分布。

可视化正是解决这些问题的工具。下面按照"图结构→度量矩阵→多维散点"的顺序，介绍三类核心可视化方法。

## 6.6.2 基于图的可视化方法

将关联规则映射为图（graph）是最直观的可视化策略之一。根据建模粒度的不同，可以定义两种图模型。

{% block type="definition" title="定义 6.23　关联规则有向图" %}
给定关联规则集合 $\mathcal{R} = \{r_1, r_2, \ldots, r_m\}$，其中每条规则 $r_i: A_i \to B_i$，**关联规则有向图** $G_R = (V, E, w)$ 定义为：

- **节点集** $V$：所有在规则中出现的项，$V = \bigcup_{i=1}^{m} (A_i \cup B_i)$；
- **有向边集** $E$：对每条规则 $r_i: A_i \to B_i$，从 $A_i$ 中的每个项 $a$ 向 $B_i$ 中的每个项 $b$ 添加有向边 $(a, b)$；
- **边权函数** $w: E \to \mathbb{R}^+$：通常取规则的某个兴趣度量（如提升度）作为边权。

节点的**加权度中心性**定义为：

$$
C_w(v) = \sum_{(v, u) \in E} w(v, u) + \sum_{(u, v) \in E} w(u, v) \tag{6.44}
$$

$C_w(v)$ 越大，表示项 $v$ 在关联网络中的"枢纽"作用越强。
{% endblock %}

关联规则有向图的一个重要特征是它保留了规则的方向性——从前件到后件的箭头方向对应着"如果…则…"的因果暗示。但当规则数量较大时，有向图可能变得非常密集，难以辨识。此时可以退一步，使用无向的项共现网络。

{% block type="definition" title="定义 6.24　项共现网络" %}
给定频繁项集集合 $\mathcal{F}$，**项共现网络** $G_C = (V, E_C, \phi)$ 定义为：

- **节点集** $V$：所有频繁项；
- **无向边集** $E_C$：对每对项 $(a, b)$，若存在频繁 2-项集 $\{a, b\} \in \mathcal{F}$，则添加无向边 $\{a, b\}$；
- **边权函数** $\phi$：取项对的某个关联度量作为权重，常用选择包括：

$$
\phi_{\text{lift}}(a, b) = \frac{P(a, b)}{P(a) \cdot P(b)}, \quad \phi_{\text{kulc}}(a, b) = \frac{1}{2}\left(\frac{P(a,b)}{P(a)} + \frac{P(a,b)}{P(b)}\right) \tag{6.45}
$$

节点大小通常映射为该项的支持度 $P(v)$，边的粗细映射为权重 $\phi$。
{% endblock %}

项共现网络的优势在于简洁——它忽略规则方向，专注于"哪些项经常一起出现"。通过社区检测算法（如 Louvain），还可以自动识别紧密关联的商品群组，这在货架摆放优化中非常有用。

在可视化实践中，两种图模型的选择取决于分析目标：

| 特征 | 关联规则有向图 | 项共现网络 |
|------|-------------|-----------|
| 边的方向 | 有向（前件→后件） | 无向 |
| 适用规则数 | ≤ 50 条 | ≤ 200 对 |
| 展示重点 | 规则方向与强度 | 项的共现结构 |
| 社区检测 | 不适用 | 适用 |
| 典型应用 | 规则解读 | 货架分区 |

## 6.6.3 基于矩阵的可视化方法

当需要全局概览所有项对之间的关联强度时，矩阵热力图（heatmap）是一种高效的可视化方式。

{% block type="definition" title="定义 6.25　关联强度矩阵" %}
给定 $n$ 个频繁项 $\{v_1, v_2, \ldots, v_n\}$，**关联强度矩阵** $M \in \mathbb{R}^{n \times n}$ 定义为：

$$
M_{ij} = \begin{cases}
\mu(v_i, v_j) & \text{if } \{v_i, v_j\} \in \mathcal{F} \\
0 & \text{otherwise}
\end{cases} \tag{6.46}
$$

其中 $\mu$ 为选定的兴趣度量（如提升度、Kulczynski 或余弦相似度）。对于有向规则 $v_i \to v_j$，矩阵为非对称矩阵：$M_{ij} = \text{conf}(v_i \to v_j)$。
{% endblock %}

矩阵热力图的最大优势在于**全局一览性**——分析师一眼就能识别高关联区域（深色块）和低关联区域（浅色块）。如果对行列进行层次聚类排序，还能自动揭示商品的聚类结构。

矩阵可视化的实践要点：

1. **对称矩阵**：使用提升度或 Kulczynski 作为度量时，$M$ 是对称的，只需显示上三角或下三角。
2. **非对称矩阵**：使用置信度时，$\text{conf}(A \to B) \neq \text{conf}(B \to A)$，需要完整矩阵。
3. **行列排序**：随机排序的矩阵往往看起来杂乱无章。使用层次聚类对行列进行重排，可以让高关联的项对聚集在一起，形成清晰的"块状结构"。
4. **颜色映射**：推荐使用发散色标（如 `RdYlBu_r`），以 $\mu = 1$（独立基线）为中心，正关联为暖色，负关联为冷色。

## 6.6.4 多维度量空间可视化

每条关联规则至少携带三个度量维度：支持度（普遍性）、置信度（可靠性）和提升度（新颖性）。在三维空间中同时展示这些度量，可以帮助分析师识别不同类型的规则。

### 6.6.4.1 支持度-置信度散点图

最经典的规则可视化方式是支持度-置信度散点图。将每条规则映射为一个点，$x$ 轴为支持度，$y$ 轴为置信度，颜色或大小映射提升度：

$$
\text{可视映射：} \quad (x, y, \text{color}, \text{size}) = (\text{supp}, \text{conf}, \text{lift}, |\text{rule}|) \tag{6.47}
$$

其中 $|\text{rule}|$ 为规则涉及的项数（前件 + 后件的总项数）。这种映射将四个维度的信息压缩到二维平面上。

分析师通常关注散点图的**右上角**——高支持度且高置信度的规则最有商业价值。但提升度的颜色映射可以揭示一些隐藏的信息：即使支持度不高，某些规则可能具有极高的提升度，表示强烈的正关联。

### 6.6.4.2 平行坐标图

当需要同时比较 4 个以上度量时，平行坐标图（parallel coordinates plot）比散点图更有效。每个度量对应一根平行竖轴，每条规则表示为一条连接各轴上对应值的折线。

平行坐标图的优势在于：
- 可以同时展示支持度、置信度、提升度、Kulczynski、全置信度等任意多个度量；
- 通过折线的"交叉模式"可以快速识别各度量之间的一致性或分歧；
- 支持交互式刷选（brushing）——在某个轴上选定区间，高亮满足条件的规则。

### 6.6.4.3 气泡图矩阵

对于同时比较多组规则（如不同算法、不同参数下发现的规则）的场景，气泡图矩阵是一种紧凑的可视化方式：每一行代表一条规则，每一列代表一个度量，气泡的大小和颜色分别编码度量值。这正是 §6.4 图 6.4.1 采用的方式。

## 6.6.5 规则过滤与冗余消除

可视化帮助我们"看见"规则，但在将规则提交给业务团队之前，还需要进行**过滤与简化**。一个常见问题是规则冗余：如果已经知道"面包→牛奶"成立，那么"面包, 鸡蛋→牛奶"可能只是前者的"特化"，不提供额外信息。

{% block type="definition" title="定义 6.26　冗余规则" %}
设规则 $r_1: A_1 \to B_1$ 和 $r_2: A_2 \to B_2$，若满足以下条件，则称 $r_2$ 关于 $r_1$ 是**冗余的**：

1. **后件相同**：$B_1 = B_2$；
2. **前件包含**：$A_1 \subset A_2$（$r_2$ 的前件是 $r_1$ 前件的超集）；
3. **置信度不增**：$\text{conf}(r_2) \leq \text{conf}(r_1)$。

记为 $r_2 \preceq_{\text{red}} r_1$。**最小非冗余规则集** $\mathcal{R}^*$ 是满足以下条件的最小子集：

$$
\mathcal{R}^* = \{r \in \mathcal{R} : \nexists\, r' \in \mathcal{R}, r \neq r', r \preceq_{\text{red}} r'\} \tag{6.48}
$$

即 $\mathcal{R}^*$ 中没有规则可以被同集合中的另一条规则"吸收"。
{% endblock %}

直觉上，如果添加更多前件条件并不能提高置信度，那么这些额外条件是冗余的。下面的定理给出了冗余检测的形式化准则。

{% block type="theorem" title="定理 6.12　冗余规则检测准则" %}
设 $r_1: A \to B$ 是一条关联规则，$r_2: A \cup C \to B$（其中 $C \cap A = \emptyset$, $C \cap B = \emptyset$）是其特化规则。若满足：

$$
\text{conf}(A \cup C \to B) \leq \text{conf}(A \to B) \tag{6.49}
$$

则 $r_2$ 关于 $r_1$ 冗余，可以安全删除而不损失信息。

等价地，用条件概率表示：

$$
P(B \mid A \cup C) \leq P(B \mid A) \tag{6.50}
$$

即在已知 $A$ 的条件下，增加 $C$ 的信息不能提高对 $B$ 出现的预测。

**证明**：根据冗余规则定义 6.26 的条件 (1)–(3)，$r_2$ 和 $r_1$ 后件相同（$B_1 = B_2 = B$），前件满足 $A \subset A \cup C$，且置信度条件 $\text{conf}(r_2) \leq \text{conf}(r_1)$ 正是式 (6.49)。因此三个条件均满足，$r_2 \preceq_{\text{red}} r_1$。

**信息论视角**：条件 (6.50) 等价于 $C$ 对 $B$ 的条件互信息为非正：

$$
I(C; B \mid A) = H(B \mid A) - H(B \mid A, C) \leq 0 \tag{6.51}
$$

即给定 $A$ 后，$C$ 不提供关于 $B$ 的额外信息（甚至可能引入噪声）。$\blacksquare$
{% endblock %}

在实际应用中，冗余消除的效果往往非常显著。以一个典型的超市数据集为例，500 条原始规则经冗余消除后通常可以精简到 80–150 条，减少 60%–80% 的规则数量，同时保留所有"真正独立"的关联模式。

除了冗余消除，规则过滤还包括以下常用策略：

| 过滤策略 | 准则 | 目的 |
|---------|------|------|
| 最小支持度 | $\text{supp}(r) \geq \sigma$ | 排除偶然共现 |
| 最小置信度 | $\text{conf}(r) \geq \delta$ | 排除弱预测力规则 |
| 最小提升度 | $\text{lift}(r) > 1$ | 排除负关联和独立性 |
| 冗余消除 | 定理 6.12 | 排除信息冗余规则 |
| 前件/后件限定 | $B \cap \mathcal{T} \neq \emptyset$ | 聚焦感兴趣的目标项 |
| 规则长度限制 | $|A| + |B| \leq k$ | 保持规则可理解性 |

## 6.6.6 规则解释与商业价值评估

通过可视化和过滤后，最终留下的规则需要转化为可操作的商业建议。但并非所有"统计上有趣"的规则都"商业上有用"。例如，"面包→牛奶"虽然置信度很高，但如果面包和牛奶在超市中已经放在相邻货架上，这条规则就不提供新的操作空间。

{% block type="theorem" title="定理 6.13　可操作规则的评估准则" %}
一条关联规则 $r: A \to B$ 被视为**可操作的**（actionable），需同时满足以下四个条件：

1. **统计显著性**：$\text{lift}(r) > 1$ 且通过 $\chi^2$ 检验（$p < 0.05$），排除偶然共现；
2. **实际显著性**：$\text{supp}(r) \geq \sigma_{\min}$，确保规则覆盖足够多的事务；
3. **新颖性**：$r \notin \mathcal{K}$，其中 $\mathcal{K}$ 为已知/常识性规则的集合——规则不应该是"正确但无用"的；
4. **可干预性**：存在可行的业务动作 $\alpha$（如调整货架位置、捆绑促销），使得执行 $\alpha$ 能利用 $r$ 揭示的关联来提升目标指标。

形式化地，规则 $r$ 的**可操作性评分**定义为：

$$
\text{ActionScore}(r) = \text{lift}(r) \cdot \text{supp}(r) \cdot \mathbb{1}[r \notin \mathcal{K}] \cdot \mathbb{1}[\exists \alpha] \tag{6.52}
$$

其中 $\mathbb{1}[\cdot]$ 为指示函数。$\text{ActionScore}(r) > 0$ 的规则才值得提交给业务团队。

**证明**（必要性论证）：

- 若条件 1 不满足（$\text{lift} \leq 1$），则 $A$ 和 $B$ 独立或负关联，不存在正向关联可利用。
- 若条件 2 不满足（$\text{supp}$ 过低），即使关联很强，受影响的客户也太少，ROI 不足。
- 若条件 3 不满足（规则为常识），则业务团队已知该关联，规则不提供新信息。
- 若条件 4 不满足（无可行动作），则规则虽有洞察但无法转化为商业价值。

四个条件中任何一个为零都使 $\text{ActionScore} = 0$。$\blacksquare$
{% endblock %}

下表将规则分类为四种典型的商业应用场景：

| 应用场景 | 规则特征 | 典型操作 |
|---------|---------|---------|
| 捆绑促销 | 高 lift，中等 supp | 联合折扣、买 A 送 B |
| 货架摆放 | 高 supp，中等 lift | 相邻摆放互补品 |
| 交叉推荐 | 高 conf，任意 supp | "买了 A 的人还买了 B" |
| 库存管理 | 高 supp，强时序性 | 联合补货策略 |

# ── 1. 生成合成购物篮数据 ──────────────────────────────────
np.random.seed(42)
items = ['面包', '牛奶', '鸡蛋', '啤酒', '薯片', '可乐',
         '酸奶', '水果', '蔬菜', '肉类']
n_trans = 2000
transactions = []
for _ in range(n_trans):
    basket = set()
    if np.random.random() < 0.35: basket.update(['面包', '牛奶'])
    if np.random.random() < 0.25: basket.update(['啤酒', '薯片'])
    if np.random.random() < 0.20: basket.update(['面包', '牛奶', '鸡蛋'])
    if np.random.random() < 0.18: basket.update(['酸奶', '水果'])
    for item, prob in [('水果',0.45),('蔬菜',0.40),('肉类',0.30),
                       ('鸡蛋',0.25),('可乐',0.20)]:
        if np.random.random() < prob: basket.add(item)
    for item in items:
        if item not in basket and np.random.random() < 0.04:
            basket.add(item)
    if basket: transactions.append(basket)
n = len(transactions)
# ── 2. 计算所有项对的度量 ──────────────────────────────────
def compute_metrics(transactions, items, n, min_support=0.02):
    """计算所有满足最小支持度的项对的关联度量"""
    item_support = {}
    for item in items:
        item_support[item] = sum(1 for t in transactions if item in t) / n
    rules = []
    for a, b in combinations(items, 2):
        pAB = sum(1 for t in transactions if a in t and b in t) / n
        pA, pB = item_support[a], item_support[b]
        if pAB < min_support or pA == 0 or pB == 0:
            continue
        conf_ab = pAB / pA  # A -> B
        lift_ab = pAB / (pA * pB)
        kulc = 0.5 * (pAB/pA + pAB/pB)
        rules.append({'antecedent': a, 'consequent': b,
                      'support': pAB, 'confidence': conf_ab,
                      'lift': lift_ab, 'kulczynski': kulc})
        conf_ba = pAB / pB  # B -> A
        rules.append({'antecedent': b, 'consequent': a,
                      'support': pAB, 'confidence': conf_ba,
                      'lift': lift_ab, 'kulczynski': kulc})
    return rules, item_support
rules, item_support = compute_metrics(transactions, items, n)
# ── 3. 冗余规则消除 ────────────────────────────────────────
def remove_redundant(rules, min_conf=0.3, min_lift=1.0):
    """过滤低质量规则并消除冗余（定义 6.26）"""
    filtered = [r for r in rules
                if r['confidence'] >= min_conf and r['lift'] >= min_lift]
    # 按后件分组，在同后件规则中消除冗余
    by_consequent = defaultdict(list)
    for r in filtered:
        by_consequent[r['consequent']].append(r)
    non_redundant = []
    for cons, group in by_consequent.items():
        group.sort(key=lambda r: -r['confidence'])
        kept = []
        for r in group:
            is_redundant = any(k['confidence'] >= r['confidence'] for k in kept)
            if not is_redundant:
                kept.append(r)
        non_redundant.extend(kept)
    return non_redundant
clean_rules = remove_redundant(rules)
# ── 4. 构建关联规则有向图（定义 6.23）─────────────────────
def build_rule_graph(rules, item_support):
    """构建关联规则有向图"""
    G = nx.DiGraph()
    for item, supp in item_support.items():
        G.add_node(item, support=supp)
    for r in rules:
        G.add_edge(r['antecedent'], r['consequent'],
                   weight=r['lift'], confidence=r['confidence'])
    return G
G = build_rule_graph(clean_rules, item_support)
# ── 5. 构建关联强度矩阵（定义 6.25）──────────────────────
def build_association_matrix(rules, items):
    """构建提升度矩阵"""
    idx = {item: i for i, item in enumerate(items)}
    M = np.ones((len(items), len(items)))
    for r in rules:
        i, j = idx[r['antecedent']], idx[r['consequent']]
        M[i, j] = r['lift']
    return M
M = build_association_matrix(rules, items)
